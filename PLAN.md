# Coin Autopilot — BTC 흐름 기반 자동매매 플랫폼

## 1. 프로젝트 개요

### 1.1 비전

BTC 흐름을 선행 신호로 활용하여 알트코인에서 수익을 추구하는 자동매매 플랫폼.
업비트(현물)와 OKX(선물)를 운용하며, 백테스팅 → 가상매매 → 실전매매를 단계적으로 전환한다.

**핵심 차별화**: 투명한 웹 기반 시그널 페이지. 텔레그램 시그널방과 달리 각 시그널의 근거(z-score, RSI, BTC 레짐 상태)를 공개. 공개 페이지(인증 불필요) + 로그인 시 개인 백테스팅/가상매매 대시보드.

### 1.2 핵심 목표

| 목표 | 설명 |
|------|------|
| **안전 우선** | MDD 제한, 레짐 필터, 포지션 사이징으로 "잃지 않는" 방향 |
| **전략 검증** | 백테스팅 → 가상매매 → 실전 3단계 파이프라인 |
| **투명한 시그널** | 공개 시그널 페이지에 매매 근거 투명하게 공개 |
| **실시간 모니터링** | 웹 대시보드에서 자산/손익/전략 상태 확인 |
| **다중 전략 비교** | 여러 전략을 동시 가상매매하여 최적 전략 선별 |
| **확장성** | 전략 플러그인 구조, 타임프레임 설정 가능 |

### 1.3 거래소

| 거래소 | 용도 | 마켓 | 비고 |
|--------|------|------|------|
| **업비트** | 현물 매매 | KRW 마켓 | JWT 인증, 초당 8회 제한 |
| **OKX** | 선물(무기한) 매매 | USDT 페어 | HMAC-SHA256, 초당 20회 |

- 공개 페이지는 React Query로 폴링 (4시간 refetch), 실시간 구독 불필요
- VPS → Supabase는 `SUPABASE_SERVICE_ROLE_KEY`로 쓰기 (RLS 우회)

## 3. 매매 전략

### 3.1 전략 플러그인 구조

모든 전략은 `Strategy` 인터페이스를 구현. 타임프레임은 전략별 설정 가능 (하드코딩 금지).

```typescript
interface Strategy {
  config: {
    id: string
    name: string
    timeframe: '1h' | '4h' | '1d'  // 설정 가능
    exchange: 'upbit' | 'okx'
    params: Record<string, number>
  }
  evaluate(candles, regime): StrategySignal[]
  evaluateExits(candles, regime, positions): ExitSignal[]
}
```

나중에 모멘텀 전략, 볼륨 스파이크 전략, 알트 독립 펌핑 감지 전략 등을 같은 인터페이스로 추가 가능.

### 3.2 전략 목록

#### 전략 1: BTC 레짐 필터 + 알트 평균회귀 (구현 완료)

> BTC가 리스크-온을 확인해준 뒤, 알트가 BTC 대비 눌린 상태를 매수

**BTC 레짐 판단 (4H 기준)**
```
Risk-On 조건 (모두 충족):
  ① BTC Close > EMA(200, 4H)
  ② BTC RSI(14, 4H) ∈ [52, 70]
  ③ BTC ATR%(14, 4H) ≤ 4.5%

Risk-Off 조건 (하나라도 충족):
  ① BTC Close < EMA(200, 4H)
  ② BTC ATR%(14, 4H) ≥ 6.5%
  ③ BTC RSI(14, 4H) ≤ 45

경계값: EMA와 가격이 같으면 이전 레짐 유지
```

**알트 진입/청산**
```
비율:    R_i = ln(ALT_i / BTC)
z-score: z_i = (R_i − SMA_20(R_i)) / STD_20(R_i)

진입 (Risk-On + 모두 충족):
  ① z_i ≤ −1.0
  ② ALT RSI(14, 4H) ≤ 78
  ③ 동시 보유 ≤ 5종목

청산 (우선순위):
  0. 손절: 진입가 − 2.7 × ATR(14, 4H) 이탈 → 즉시
     레짐 스톱: BTC Risk-Off → 전체 포지션 청산
  1. 이익 실현: z_i ≥ 0.0
  2. 시간 청산: 8캔들(32시간) 경과
```

#### 전략 2: BTC 도미넌스 로테이션 (미구현)

> BTC.D 하락 + USDT.D 하락 = 알트 시즌 포착

#### 전략 3: BTC 변동성 타이밍 (미구현)

> 변동성 낮고 추세 양호 → 알트 익스포저 확대

#### 전략 4: 펀딩비 차익 (미구현, OKX 선물 전용)

> 펀딩비 극단 시 델타중립 포지션

#### 전략 5: 김프 모니터링 (미구현)

> 업비트-OKX 간 가격 괴리 추적

### 3.3 투자 성향 프로필

| 프로필 | 레버리지 | 동시 보유 | MDD 한도 | ATR 손절 | 전략 |
|--------|----------|-----------|----------|----------|------|
| **안전** | 1x (현물만) | 3종목 | 10% | 2.0×ATR | 전략1 단독 |
| **중립** | 1~2x | 5종목 | 15% | 2.7×ATR | 전략1 + 전략3 |
| **공격** | 2~3x | 8종목 | 25% | 3.5×ATR | 전략1~4 조합 |

### 3.4 포지션 사이징 & 리스크 관리

```
일일 최대 손실   = Equity × 2%
단일 포지션 손실 = Equity × 0.30%
수량 계산        = (Equity × 0.003) / (STOP_ATR_MULT × ATR)
동시 보유 상한   = 프로필별 3~8종목

MDD 15% 초과 → 리스크 50% 축소
MDD 25% 초과 → 매매 중단 + 알림

선물 추가 규칙:
  - 마진 모드: 격리(Isolated)
  - 지갑 증거금 사용률 < 35%
  - 모든 청산 주문: reduce-only
  - 레버리지 상한: 프로필별 1~3x
```

### 3.5 알트코인 유니버스

현재 대상 (업비트 KRW 마켓):
**BTC, ETH, XRP, SOL, DOGE, ADA, AVAX, DOT, LINK, ATOM**

선정 기준:
- 30일 평균 일거래대금 ≥ 50억 KRW
- 스프레드 ≤ 20bp
- 스테이블코인/랩드/레버리지 토큰 제외
- 신규 상장 14일 이내 제외

## 4. 백테스팅 시스템

### 4.1 간이 백테스트 (구현 완료)

| 항목 | 설정 |
|------|------|
| **데이터** | 최근 6개월 4H 캔들 (Upbit REST API) |
| **수수료** | 업비트 0.05% (매수/매도 각각) |
| **슬리피지** | 다음 4H 캔들 시가 기준 ±0.1% |
| **체결** | 시그널 발생 → 다음 캔들 시가에 체결 |
| **워크포워드** | 미적용 (본격 백테스트에서 추가) |
| **생존편향** | 미제거 (본격 백테스트에서 추가) |

성과 지표: Sharpe Ratio, MDD, 승률, 총 거래수, CAGR, 평균 보유시간

### 4.2 본격 백테스트 (미구현, TODOS 참조)

- 생존편향 제거: 과거 시점별 거래 가능 유니버스 재구성
- 워크포워드: 70% 학습 / 30% 검증, 롤링 윈도우
- 스트레스 테스트: 차이나 크래시, LUNA, FTX, 엔캐리 등

#### 공통 상태 패턴

**로딩 (스켈레톤):**
- 카드: `var(--surface)` 배경 + shimmer 애니메이션 (2초 사이클, 좌→우, infinite, ease-in-out)
- 테이블: 3행 스켈레톤, 열 너비 비율 유지
- 숫자: `----` placeholder (JetBrains Mono)
- 차트: 빈 프레임 + 중앙 `Loader2` 스피너

**부분 로드 (디자인 리뷰 추가):**
- 각 섹션(KPI, 차트, 포지션, 레짐 등)이 독립적으로 로드/스켈레톤 표시
- 먼저 로드된 섹션부터 즉시 표시, 나머지는 스켈레톤 유지
- 페이지 전체 로딩 스피너 금지 — 섹션별 독립 렌더링

**에러 에스컬레이션 (디자인 리뷰 추가):**
- **Tier 1 (인라인 재시도):** 개별 섹션 실패 — `AlertTriangle` + 메시지 + "다시 시도"
- **Tier 2 (토스트):** 인증/핵심 경로 실패 — 화면 하단 토스트, `--loss` 배경
- **Tier 3 (모달):** 인프라 다운 (Supabase 연결 끊김) — 중앙 모달, 자동 재연결 시도
- 전체 페이지 에러 금지 — 실패한 섹션만 에러 표시, 나머지 정상 유지
- 색상: `--loss` 아이콘, `--text-secondary` 메시지

**빈 상태:**
- 중앙 정렬, `1px dashed var(--border)` 테두리

### 7.8 온보딩 흐름

> 최초 로그인 사용자가 빈 대시보드를 보고 이탈하지 않도록.
> **디자인 리뷰 결정:** 환영 모달 제거. CTA 경로와 대시보드 체크리스트로 대체. 마찰 최소화.

#### CTA 경로 (시그널 페이지에서 진입)

```
시그널 CTA "직접 백테스트 해보기 →" 클릭
     │
     ▼
로그인 모달 (매직 링크 / GitHub OAuth)
     │
     ▼
[환영 모달 스킵 — CTA 경로에서는 온보딩 생략]
     │
     ▼
빠른 백테스트 시트 (인라인, 페이지 상단)
"BTC 평균회귀 전략으로 백테스트 실행?"
[▶ 실행] [커스터마이즈]
     │
     ▼
결과 표시 + "이 전략으로 가상매매 시작" CTA
```

#### 일반 경로 (직접 로그인)

```
로그인 → 대시보드 (빈 상태)
     │
     ▼
빈 상태 = 통합 온보딩 체크리스트 (별도 모달 아님)
┌─────────────────────────────────┐
│ Coin Autopilot 시작하기         │
│                                 │
│ [○] 1. 투자 성향 선택           │
│     [전략 설정으로 이동 →]       │
│                                 │
│ [○] 2. 첫 백테스트 실행         │
│     [백테스트 실행 →]            │
│                                 │
│ [○] 3. 가상매매 시작            │
│     [2단계 완료 후 활성화]       │
│                                 │
│ [닫기]                          │
└─────────────────────────────────┘
```

**온보딩 설계 원칙:**
- **환영 모달 제거** — CTA 경로에서는 불필요한 마찰, 일반 경로에서는 대시보드 빈 상태로 대체
- 대시보드 빈 상태 = 온보딩 체크리스트 (별도 컴포넌트 아님, 빈 상태 자체가 온보딩)
- "닫기(dismiss)" 가능 — 닫으면 대시보드 빈 상태로 전환
- 각 단계 CTA가 해당 페이지로 직접 연결
- 온보딩 완료 플래그는 Supabase user metadata에 저장

### 7.9 사용자 여정 스토리보드

```
1. 발견 (비로그인)
   시그널 페이지 → "이 시그널, 근거가 다 보이네?" → 신뢰 형성
   ↓
2. 관심 (비로그인)
   백테스트 성과 확인 → "Sharpe 1.42면 괜찮은데?" → CTA "직접 해보기"
   ↓
3. 전환 (로그인)
   로그인 → 온보딩 → 투자 성향 선택 → 첫 백테스트
   ↓
4. 활성화 (가상매매)
   가상매매 시작 → 매일 포지션 확인 → "실제로 돈 벌 수 있겠는데"
   ↓
5. 유지 (실전)
   실전매매 전환 → 대시보드 모니터링 → 전략 비교/조정
   ↓
6. 확장
   새 전략 추가 → 다중 전략 운용 → 고급 파라미터 조정
```

**각 단계 감정 목표:**
- 발견: "투명하다, 신뢰할 수 있다"
- 관심: "성과가 괜찮다, 직접 확인하고 싶다"
- 전환: "쉽다, 막히는 곳이 없다"
- 활성화: "실제 시장과 동일하게 움직인다"
- 유지: "내 자산이 잘 관리되고 있다"
- 확장: "더 좋은 전략을 찾을 수 있다"

**이탈 방지 포인트:**
| 이탈 지점 | 원인 | 해결 |
|-----------|------|------|
| 시그널 페이지 → 이탈 | 근거가 안 보임 | 근거를 카드에 인라인 표시 |
| 로그인 후 → 이탈 | 빈 대시보드 | 온보딩 체크리스트 |
| 백테스트 후 → 이탈 | 다음 단계 불명확 | "가상매매 시작" CTA |
| 가상매매 중 → 이탈 | 성과 확인 번거로움 | 대시보드 KPI 자동 반영 |

## 8. DB 스키마

```sql
candles (exchange, symbol, timeframe, OHLCV, UNIQUE 복합키)
signals (strategy, symbol, direction, z_score, rsi, btc_regime, reasoning jsonb, is_active)
regime_states (regime, btc_close, ema_200, rsi_14, atr_pct)
backtest_results (user_id, strategy, params jsonb, sharpe, mdd, win_rate, equity_curve jsonb)
positions (user_id, session_type, strategy, symbol, pnl, status)

RLS:
  candles, signals, regime_states → anon SELECT 허용 (공개)
  backtest_results, positions → auth.uid() = user_id (개인)
  VPS 서버 → service_role key (RLS 우회)
```
