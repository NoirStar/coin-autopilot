# Coin Autopilot 현재 구현 분석

작성 기준: 2026-04-01, 현재 저장소의 실제 코드만 기준으로 정리했다.  
`README.md`, `PLAN.md`, `DESIGN.md`의 목표 상태가 아니라 `/server`, `/web`, `/supabase`에 있는 현재 구현 상태를 요약한 문서다.

## 1. 한눈에 보는 현재 상태

- 지금 프로젝트는 "BTC 레짐 기반 분석 + 알트 시그널/탐지 + 백테스트 + 가상매매 + 부분적 실전매매 엔진"까지는 갖춰져 있다.
- 가장 구현이 진척된 영역은 서버 쪽 전략 로직, 백테스트 엔진, 탐지 스캔, 크론 기반 자동 파이프라인이다.
- 프론트는 화면 수는 많지만, 일부 화면은 실제 데이터 연결보다 운영용 껍데기나 플레이스홀더에 가깝다.
- README에 적힌 전략/구성과 실제 구현은 다르다. 특히 도미넌스 로테이션, 펀딩비 차익, 김프 모니터링은 현재 핵심 전략으로 구현되어 있지 않다.
- "실전 자동매매 플랫폼"처럼 보이지만, 실제 라이브 실행 범위는 OKX 선물 일부 전략으로만 제한되어 있다.

## 2. 현재 구조

### 서버

- 기술: Node.js + TypeScript + Hono + Supabase + node-cron + ccxt
- 진입점: `server/src/index.ts`
- 역할:
  - 공개 API: 대시보드 요약, 시그널, 백테스트 결과, 알트 탐지
  - 인증 API: 전략 관리, 가상매매, 포트폴리오, 설정
  - 크론: 4시간 시그널/가상매매, 1시간 탐지 스캔, 일일 캐시 정리

### 웹

- 기술: React 19 + Vite + React Query + Zustand + Supabase Auth
- 진입점: `web/src/App.tsx`
- 화면 구조:
  - 공개 화면: `/signals`, `/detection`
  - 운용자 화면: `/operator/dashboard`, `/operator/strategy`, `/operator/backtest`, `/operator/paper-trading`, `/operator/portfolio`, `/operator/settings`

### 데이터 저장

- 기본 스키마: `server/src/db/schema.sql`
- 추가 마이그레이션: `supabase/migrations/20260328_detection_cache.sql`
- 핵심 테이블:
  - `candles`
  - `regime_states`
  - `signals`
  - `backtest_results`
  - `positions`
  - `strategies`
  - `user_settings`
  - `paper_sessions`
  - `detection_cache`

## 3. 실제 구현된 전략

### 공통 상위 로직

- 모든 전략 위에 `BTC 레짐 필터`가 존재한다.
- 레짐 판정 기준:
  - Risk-On: `BTC > EMA200` AND `RSI 52~70` AND `ATR% <= 4.5`
  - Risk-Off: `BTC < EMA200` OR `ATR% >= 6.5` OR `RSI <= 45`
- 이 레짐은 시그널 생성, 백테스트, 가상매매 엔진에서 공통으로 사용된다.

### 1. 알트 평균회귀

- ID: `alt_mean_reversion`
- 거래소/타임프레임: Upbit 현물, 4H
- 아이디어:
  - BTC가 Risk-On일 때
  - BTC 대비 많이 눌린 알트코인(z-score 하락)을 매수
- 진입:
  - `zScore <= -1.0`
  - `RSI <= 78`
- 청산:
  - z-score 평균 복귀
  - ATR 기반 손절
  - 시간 청산
  - 레짐 Risk-Off 전환 시 전체 청산
- 현재 사용처:
  - 시그널 생성 O
  - 백테스트 O
  - 가상매매 O
  - 실전매매 X

### 2. BTC EMA 크로스오버

- 대상: BTC, ETH

  - EMA(12/26) 교차

- 청산:

## 4. 현재 구현된 기능

### 4.1 시그널 생성 파이프라인

- 크론 주기: 4시간
- 동작:
  - BTC 캔들 직접 조회
  - Upbit KRW 알트 상위 20개 조회
  - BTC 레짐 계산
  - 전략별 시그널 생성
  - `signals`, `regime_states` 저장
  - 오래된 시그널 비활성화
- 비고:
  - Upbit는 현물 알트, OKX는 BTC/ETH 선물 위주
  - 탐지 전략은 이 시그널 파이프라인에 포함되지 않음

### 4.2 알트코인 탐지

- 크론 주기: 1시간
- 동작:
  - Upbit KRW 전체 마켓 스캔
  - BTC 1H 흐름과 각 코인 1H 캔들/호가를 결합
  - 복합 점수 계산
  - `detection_cache` 저장
  - `score >= 0.8` 강한 종목은 텔레그램 알림
- 사용자 인터페이스:
  - 수동 스캔
  - SSE 진행률 표시
  - 캐시 기반 재조회
  - 결과 카드 펼침 상세

### 4.3 백테스트

- 동기 실행과 SSE 스트리밍 실행 둘 다 존재
- 캔들 캐시:
  - DB에 있으면 재사용
  - 없거나 오래됐으면 거래소 API에서 가져와 upsert
- 결과:
  - Sharpe, 승률, MDD, CAGR, 거래 수, 평균 보유시간, 에퀴티 커브
  - `backtest_results` 저장
- 특징:
  - 롱/숏
  - 수수료
  - 슬리피지
  - 레버리지
  - 트레일링
  - 다음 캔들 시가 체결 가정

### 4.4 가상매매

- `paper_sessions` 기반 다중 세션 구조
- 세션 생성/일시정지/재개/종료 API 존재
- 4시간 크론마다:
  - 전략 평가
  - DB `positions` 기준 진입/청산
  - 세션 성과 재계산
- 현재 구현 범위:
  - 서버 로직은 실제로 동작
  - UI는 세션 관리까지는 연결되어 있음
  - 하지만 포지션/체결 내역 화면은 아직 빈 상태 메시지 위주

### 4.5 실전매매

- 환경변수 `LIVE_TRADING=true`일 때만 실행
- 대상 거래소: OKX 선물
- 현재 라이브 전략 지원:
  - `btc_ema_crossover`
  - `btc_bollinger_reversion`
- 안전장치:
  - 일일 최대 손실 한도
  - 포지션 크기 계산
  - isolated 마진
  - stop-market 주문
  - 전략 간 충돌 방지
- 상태:
  - 엔진 코드는 있음
  - UI에서 직접 운용하는 화면은 없음
  - 실제 운영용이라고 보기엔 아직 내부 엔진 성격이 강함

### 4.6 인증과 사용자 설정

- Supabase Auth 사용
- 로그인 방식:
  - 이메일 매직 링크
  - GitHub OAuth
  - Google OAuth
- 설정:
  - 리스크 프로필
  - API 키 저장/삭제
  - 알림 on/off
  - 서버 상태 조회

### 4.7 데이터 수집/운영 스크립트

- 캔들 backfill 스크립트 존재
- 테스트 파이프라인 스크립트 존재
- Upbit/OKX API 직접 조회 + Supabase 캐시 구조

  - 서버 상태
  - API 키 관리
  - 리스크 파라미터
  - 알림 on/off
  - 데이터 관리 영역
- 현재 상태:
  - API 키 저장/삭제는 연결됨
  - 리스크 파라미터 저장도 연결됨
  - 알림은 채널 on/off 및 조건 토글 정도만 있음
  - 데이터 관리 영역은 확인 UI만 있고 실제 삭제 로직은 없음
  - 서버 상태 응답은 하드코딩 값이 섞여 있어 운영 현황이라기보다 상태 배지에 가깝다

## 6. 코드 기준으로 보이는 현재 한계와 어긋남

### 6.1 문서와 실제 구현의 차이

- `README.md`에는 다음이 주요 전략처럼 보이지만 실제 핵심 구현은 아님:
  - 도미넌스 로테이션
  - 펀딩비 차익
  - 김프 모니터링
- `README.md`에는 `proto/`가 있다고 적혀 있지만 현재 저장소에 `proto/` 디렉터리는 없다.

### 6.2 프론트의 데이터 흐름이 일관되지 않음

- 어떤 화면은 서버 API를 사용하고
- 어떤 화면은 브라우저에서 Supabase를 직접 조회하고
- 어떤 화면은 외부 API를 직접 두드린다.
- 그래서 현재 구조는 "서버 중심 운영 대시보드"라기보다 "프론트가 데이터 출처를 여러 군데 직접 붙인 상태"에 가깝다.

### 6.3 실전매매 범위가 UI 인상보다 좁음

- 전략 목록과 백테스트/가상매매는 5전략 기준으로 보이지만
- 실전 엔진은 현재 2전략만 실제 진입 로직에 연결되어 있다.

### 6.4 가상매매 UI가 서버 로직을 다 따라가지 못함

- 서버에는 세션, 포지션, 성과 계산이 있다.
- 하지만 화면은 아직 세션 관리 중심이고, 실제 포지션/체결 내역을 충분히 드러내지 못한다.

### 6.5 포트폴리오가 "실계좌 포트폴리오"는 아님

- `balance` API는 실제 거래소 자산이 아니라 DB 포지션 합산 기반이다.
- Upbit KRW, OKX USD 현금 잔고는 현재 0으로 내려온다.

### 6.6 상태 표시 중 일부는 사실상 하드코딩

- 하단 StatusBar의 전략 수, 포지션 수
- 설정 페이지의 서버 상태 일부 값
- 사이드바의 "서버 연결됨"
- 이런 요소는 운영 상태 표시처럼 보이지만 현재는 정적이거나 단순 응답이다.

### 6.7 스키마 정합성 이슈가 보임

- 코드가 사용하는 `positions` 관련 필드:
  - `session_id`
  - `peak_price`
  - `stop_order_id`
  - `exit_reason`
- 하지만 기본 `server/src/db/schema.sql`에는 이 컬럼들이 없다.
- 즉:
  - 실제 DB에는 수동 변경이 들어갔거나
  - 누락된 마이그레이션이 있거나
  - 현재 스키마 파일이 코드 상태를 따라오지 못하고 있다.

### 6.8 프론트에 죽은 코드/덜 연결된 코드가 있음

- `PublicLayout` 컴포넌트가 있지만 실제 라우팅에는 사용되지 않는다.
- Zustand 스토어도 일부만 실제 화면에서 사용되고, 나머지는 준비만 된 상태다.
- `DashboardPage`의 코인 분석 현황 위젯은 현재 페이지 흐름상 채워지기 어려운 캐시 키를 보고 있다.

### 6.9 알림 설정 UI와 서버 기능이 완전히 맞물리진 않음

- 서버는 Telegram bot token/chat id, Discord webhook URL 저장 API를 갖고 있다.
- 하지만 현재 설정 화면은 채널 on/off 토글만 제공하고, 해당 상세 입력 필드는 없다.

## 7. 지금 이 프로젝트를 한 문장으로 정의하면

현재의 `coin-autopilot`은 "BTC 레짐 기반 분석 도구와 실험용 자동매매 엔진이 한 저장소 안에 공존하는 상태"에 가깝다.  
백엔드 전략/엔진은 생각보다 많이 구현되어 있지만, 프론트와 데이터 스키마가 그 구현 수준을 아직 일관되게 반영하지 못하고 있다.

## 8. 리팩터링 전에 특히 중요하게 봐야 할 포인트

- 서버 중심 구조로 갈지, 프론트 직접 조회 혼합 구조를 유지할지 먼저 정리 필요
- 전략 체계:
  - 공개 시그널 전략
  - 탐지 전략
  - 가상매매 전략
  - 실전매매 전략
  - 이 4개 레이어를 하나의 전략 체계로 합칠지 분리할지 결정 필요
- 스키마 정합성:
  - 코드에서 쓰는 컬럼과 실제 마이그레이션을 맞춰야 함
- 화면 목적 재정의:
  - 공개 분석 서비스인지
  - 운영자용 콘솔인지
  - 둘을 한 앱에 둘지
- 라이브 기능 범위 정리:
  - 지금처럼 일부 전략만 라이브로 둘지
  - 가상/백테스트/라이브 지원 범위를 전략별로 명확히 나눌지
