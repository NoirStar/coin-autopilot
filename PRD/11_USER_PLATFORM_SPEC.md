# 11 User Platform Spec

## 1. 문서 목적

- 현재 개인용 중심 구조를, 장기적으로 다른 사용자도 사용할 수 있는 플랫폼 구조로 어떻게 확장할지 정의한다.
- 이 문서는 구현 지시용 PRD이며, 대화 과정은 남기지 않고 `합의된 정책`만 기록한다.

## 2. 한 줄 정의

- 사용자 플랫폼 계층은 전략, 연구 결과, 페이퍼 세션, 승인, 알림, 설정을 `사용자 단위로 분리`하고, 장기적으로는 검증된 전략을 선택해 수동/반자동/자동 운용할 수 있는 멀티유저 구조를 제공한다.

## 3. 범위

### 포함

- 로그인/계정 구조
- 사용자별 데이터 분리
- 사용자별 전략 목록
- 사용자별 연구/페이퍼 세션 분리
- 전략 선택과 추천
- 수동/반자동/자동 운용 모드
- 장기적인 멀티유저 확장 방향

### 제외

- 요금제/결제 정책
- 커뮤니티 기능
- 공개 전략 마켓플레이스 운영 세부 정책
- 실전 자금 위임/법적 구조

## 4. 핵심 원칙

- 초기에는 개인 사용자 중심으로 시작한다.
- 구조는 처음부터 사용자 단위 분리가 가능해야 한다.
- 전략, 성과, 알림, 승인 상태는 사용자별로 분리 가능해야 한다.
- 공용 전략 카탈로그와 사용자별 운용 상태는 구분한다.
- 장기적으로 사용자는 전략을 선택해서 수동/반자동/자동 모드로 운용할 수 있어야 한다.

## 5. 사용자 모델

### 5.1 초기

- 사실상 1차 사용자는 본인
- 하지만 내부 구조는 `single-user hardcode`를 피한다.

### 5.2 장기 확장

- 일반 개인 사용자
- 자동매매 입문자
- 전략을 비교해서 직접 선택하고 싶은 사용자
- 수동/반자동/자동 운용을 혼합하고 싶은 사용자

## 6. 계정 구조

### 6.1 기본 정책

- 사용자 단위 계정이 존재해야 한다.
- 각 사용자 계정은 아래 리소스를 독립적으로 가진다.

### 6.2 사용자별 분리 대상

- 전략 즐겨찾기/활성화 목록
- 백테스트 요청과 결과 뷰
- 페이퍼 세션
- 승인 대기 상태
- 알림 설정
- 리스크 설정
- 대시보드 개인 설정

## 7. 데이터 분리 원칙

### 7.1 공용 데이터

- 시세 데이터
- 파생 지표
- 시장 구조 데이터
- 시스템 전략 카탈로그
- 공용 연구 결과 메타데이터

### 7.2 사용자 전용 데이터

- 사용자별 선택 전략
- 사용자별 페이퍼 세션
- 사용자별 승인 이력
- 사용자별 알림 읽음 상태
- 사용자별 리스크 설정
- 사용자별 수동/자동 운용 설정

### 7.3 반공용 데이터

- 공용 전략 성과 요약은 가능
- 그러나 사용자별 세션 세부 로그와 개인 설정은 분리

## 8. 전략 카탈로그 구조

### 8.1 시스템 전략 카탈로그

- 플랫폼에 등록된 전략 목록
- 전략 설명
- 자산군
- 전략 성격
- 위험도
- 연구/검증 상태

### 8.2 사용자 전략 선택 목록

- 사용자가 실제로 관심/활성화한 전략 목록
- 사용자는 공용 전략 카탈로그에서 전략을 선택할 수 있다.

### 8.3 장기 확장 방향

- 검증된 전략 추천
- 전략 성과 랭킹
- 전략 비교 보기
- 사용자가 전략을 복제/활성화하는 흐름

## 9. 사용자 운용 모드

### 9.1 수동 모드

- 사용자가 전략을 직접 선택한다.
- 배포, 실행, 변경 여부를 직접 판단한다.

### 9.2 반자동 모드

- 시스템이 전략 후보를 추천한다.
- 사용자가 승인 후 반영한다.

### 9.3 자동 모드

- 충분한 검증 데이터가 있는 전략군만 허용
- 장기적으로 사용자 설정에 따라 자동 배치/교체 허용

## 10. 전략 추천 구조

### 10.1 추천 기준

- 연구 루프 검증 결과
- 페이퍼트레이딩 성과
- 오케스트레이터 시장 적합도
- 리스크 프로필

### 10.2 추천 표현 방식

- 추천 전략 리스트
- 추천 이유 요약
- 위험도 표시
- 최근 검증 결과 요약

### 10.3 장기 확장

- 사용자 성향별 추천
- 보수형/공격형 전략 추천
- 자산군별 추천

## 11. 사용자별 페이퍼 세션 구조

- 각 사용자는 자신의 페이퍼 세션을 가진다.
- 같은 전략이라도 사용자별 자본, 리스크 설정, 운용 모드가 다를 수 있다.
- 따라서 세션 상태와 결과는 사용자 단위로 분리해야 한다.

## 12. 사용자별 승인 구조

### 12.1 승인 대상

- 페이퍼 배포
- 전략 교체
- 자동화 확대
- 실전 후보 승격

### 12.2 승인 저장 항목

- user_id
- approval target
- reason
- requested_at
- approved_at
- rejected_at

## 13. 사용자별 알림 구조

- 텔레그램/디스코드 연결 여부는 사용자별 설정
- 알림 우선순위 필터도 사용자별로 가질 수 있다.
- 읽음/미읽음 상태도 사용자별로 저장한다.

## 14. 사용자별 대시보드 개인화

사용자는 아래를 개인화할 수 있어야 한다.

- 홈에서 먼저 볼 패널
- 알림 필터
- 자산군별 보기
- 수동/자동 모드 기본값
- 전략 즐겨찾기

## 15. 멀티유저 확장 시 오케스트레이터 영향

- 공용 연구 결과는 여러 사용자가 참고할 수 있다.
- 하지만 실제 전략 배치와 세션 상태는 사용자별로 다르다.
- 따라서 오케스트레이터는 `공용 전략 후보 집합`과 `사용자별 배치 상태`를 분리해서 다뤄야 한다.

## 16. 멀티유저 확장 시 리스크 영향

- 리스크 한도는 사용자별로 달라질 수 있다.
- 같은 전략도 사용자 A는 보수적, 사용자 B는 공격적으로 운용할 수 있다.
- 따라서 리스크 정책은 전략 기본값과 사용자 override를 분리해야 한다.

## 17. 장기 제품 방향

### 17.1 개인 도구 단계

- 본인 위주
- 구조적 분리만 준비

### 17.2 제한적 멀티유저 단계

- 로그인
- 사용자별 세션/알림/설정 분리
- 전략 선택 및 추천

### 17.3 플랫폼 단계

- 검증 전략 카탈로그
- 전략 성과 랭킹
- 사용자별 운용 모드
- 장기적으로 플랫폼화

## 18. 성공 기준

- 사용자 단위로 데이터 분리 가능한 구조를 가진다.
- 전략 카탈로그와 사용자 운용 상태를 구분한다.
- 사용자별 페이퍼 세션, 알림, 승인 상태를 분리 저장할 수 있다.
- 장기적으로 다른 사용자가 들어와도 아키텍처를 다시 뜯지 않아도 된다.

## 19. 오픈 질문

- 초기 로그인은 이메일 기반으로만 갈지
- 사용자별 전략 복제 기능을 둘지
- 공용 전략 성과를 어디까지 사용자에게 노출할지
- 장기적으로 사용자 간 전략 비교를 허용할지
