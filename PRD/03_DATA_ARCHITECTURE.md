# 03 Data Architecture

## 1. 문서 목적

- 오케스트레이터, 자동 연구 루프, 백테스트, 페이퍼트레이딩, 대시보드가 공통으로 사용할 데이터 구조를 정의한다.
- 데이터 수집, 정규화, 파생 지표 생성, 저장 계층, 보관 정책을 하나의 기준으로 정리한다.
- 이 문서는 구현 지시용 PRD이며, 대화 과정은 남기지 않고 `합의된 결정사항`만 기록한다.

## 2. 범위

### 포함

- 암호화폐 데이터 구조
- 한국주식 데이터 구조
- 거래소/브로커별 수집 계층
- 원본 데이터, 정규화 데이터, 파생 지표, 연구 결과, 런타임 상태 저장 구조
- 심볼 표준화
- 시간 기준
- 결측 처리 방향
- 고용량 데이터 보관 방향

### 제외

- 개별 거래소 API 명세 상세
- 주문 실행 엔진의 세부 동작
- 백테스트 엔진 내부 계산 로직
- 사용자 인증과 멀티유저 권한 구조 상세

## 3. 아키텍처 목표

- 오케스트레이터가 검증 데이터 기반으로 판단할 수 있어야 한다.
- 동일 자산을 여러 소스에서 가져와도 하나의 내부 식별 체계로 관리할 수 있어야 한다.
- 백테스트와 페이퍼트레이딩이 같은 데이터 기준을 공유해야 한다.
- 고빈도 데이터는 운영 가능성을 해치지 않는 범위에서 선택적으로 장기 보관해야 한다.
- 장기적으로 멀티브로커, 멀티거래소, 멀티유저 플랫폼으로 확장 가능해야 한다.

## 4. 시스템 배포 및 저장 토폴로지

### 4.1 Web

- `Vercel`에 배포한다.
- 역할:
  - 대시보드 렌더링
  - 전략/연구/페이퍼 결과 조회
  - 설정 및 승인 UI
  - 알림 확인 UI

### 4.2 App Server / Worker

- 초기에는 `로컬 서버/로컬 워커` 환경에서 운영한다.
- 역할:
  - 거래소/브로커 데이터 수집
  - 백테스트 작업 실행
  - 자동 연구 루프 실행
  - 페이퍼트레이딩 실행
  - 오케스트레이션 주기 작업
  - 고용량 원본 데이터 처리

### 4.3 Primary Database

- `Supabase`를 주 저장소로 사용한다.
- 역할:
  - 정규화 시계열 데이터 저장
  - 파생 지표 저장
  - 연구 결과 저장
  - 페이퍼 세션/성과 저장
  - 오케스트레이터 상태 저장
  - 승인 대기 상태 저장
  - 전략 메타데이터 및 사용자 설정 저장

## 5. 데이터 계층

### 5.1 Raw Layer

- 거래소/브로커에서 받은 원본 데이터
- 예:
  - OHLCV
  - trade tick
  - orderbook snapshot / diff
  - funding rate
  - open interest
  - long/short ratio
  - premium index
  - market warning/caution
  - 한국주식 시세/체결/호가

### 5.2 Normalized Layer

- 소스별 다른 형식을 내부 공통 형식으로 정리한 계층
- 예:
  - symbol normalization
  - timestamp normalization
  - market type normalization
  - currency normalization
  - exchange/broker mapping

### 5.3 Derived Layer

- 전략과 오케스트레이터가 직접 사용하는 계산 결과 계층
- 예:
  - EMA, RSI, MACD, Bollinger
  - volatility
  - z-score
  - funding anomaly
  - liquidity score
  - warning snapshot
  - 김치 프리미엄 / 역김치 프리미엄

### 5.4 Research Layer

- 자동 연구 루프와 백테스트 결과 저장 계층
- 예:
  - strategy run result
  - parameter set
  - walk-forward result
  - expectation
  - MDD
  - Sharpe / DSR / 보정 지표
  - trade log

### 5.5 Runtime Layer

- 오케스트레이션과 페이퍼트레이딩의 실시간 상태 계층
- 예:
  - active strategy per asset
  - current allocation
  - current positions
  - approval pending items
  - runtime alerts
  - paper session state

## 6. 자산군별 데이터 소스 방향

### 6.1 암호화폐

- 우선 거래 채널:
  - Upbit
  - OKX
- 보조 데이터 소스:
  - Binance 필요 시 사용
- 자산 범위:
  - Upbit 현물
  - OKX 선물

### 6.2 한국주식

- `broker adapter` 구조를 전제로 설계한다.
- 실제 초기 구현은 `1개 브로커`부터 시작한다.
- 의미:
  - 인터페이스는 다중 브로커 확장 가능
  - 초기 구현 속도는 유지
  - 장기적인 멀티브로커/멀티유저 확장성 확보

## 7. 시간 기준

- 내부 저장 기준은 전부 `UTC`로 통일한다.
- 화면, 알림, 사용자 설명에서만 `KST` 또는 사용자 로컬 시간으로 변환한다.

### 이유

- Upbit, OKX, Binance, 한국주식을 하나의 시간축에서 비교할 수 있다.
- 백테스트와 오케스트레이터 판단이 시간대 차이 때문에 흔들리지 않는다.
- 리샘플링과 데이터 정합성 검증이 쉬워진다.

## 8. 심볼 표준화 정책

- 내부에서는 `공통 심볼 키`를 사용한다.
- 거래소/브로커 원본 심볼은 별도 매핑 테이블로 관리한다.

### 예시

- `BTC-USDT-SWAP-OKX`
- `KRW-XRP-UPBIT`
- `KRX-005930`

### 목적

- 백테스트, 페이퍼트레이딩, 오케스트레이터, 대시보드, 알림이 같은 식별 체계를 공유한다.
- 같은 자산을 여러 데이터 소스에서 가져와도 내부적으로 일관되게 처리할 수 있다.

## 9. 저장 정책

### 9.1 장기 저장 대상

- OHLCV
- 체결 틱
- 정규화 시계열
- 파생 지표
- 연구 결과
- 페이퍼트레이딩 결과
- 오케스트레이터 상태

### 9.2 제한적 저장 대상

- 호가 데이터는 전체 장기 저장을 기본으로 하지 않는다.
- 호가는 `단기 캐시` 또는 `이벤트 구간 중심 저장`을 기본 정책으로 한다.

### 이유

- 고빈도 호가 데이터를 전부 장기 적재하면 저장 비용과 운영 부담이 급격히 커진다.
- 초기에 호가 기반 전략은 후순위다.
- 반면 체결 틱은 향후 고급 전략, 체결 품질 분석, 이벤트 리플레이에 가치가 높다.

## 10. 파생 데이터 저장 정책

다음 파생 데이터는 기본 저장 대상으로 본다.

- funding rate
- open interest
- long/short ratio
- premium index
- liquidation event proxy
- 김치 프리미엄
- 역김치 프리미엄 상태

### 목적

- 선물 특화 전략을 지원한다.
- 오케스트레이터가 가격 외 구조적 신호를 참고할 수 있게 한다.
- 나중에 "왜 이 시점에 전략이 바뀌었는가"를 재현할 수 있게 한다.

## 11. 현재 확정된 저장 계층 역할

### 11.1 Local Server / Worker

- 거래소/브로커 원본 데이터 수집
- 고빈도 체결 틱 수집
- 호가 단기 캐시
- 백테스트 작업 실행
- 자동 연구 루프 실행
- 페이퍼트레이딩 실행
- 오케스트레이션 주기 작업

### 11.2 Supabase

- 정규화된 시계열 데이터
- 파생 지표 및 계산 결과
- 전략 메타데이터
- 백테스트/워크포워드 결과
- 페이퍼 세션과 성과
- 오케스트레이터 상태
- 승인 대기 항목
- 장기 설정과 상태

### 11.3 Vercel Web

- 사용자-facing 화면
- 대시보드 조회
- 전략/연구/페이퍼 결과 조회
- 승인/알림 UI

## 12. 결측 및 고용량 데이터 처리 정책

### 12.1 Upbit 저유동성 결측 캔들 처리

- 결측 캔들은 그대로 비워두지 않는다.
- 이전 종가 기준으로 빈 캔들을 생성한다.
- 동시에 `no_trade=true` 또는 이에 준하는 상태 플래그를 저장한다.

### 이유

- 시계열 연속성을 유지할 수 있다.
- 지표 계산기와 백테스트 엔진이 안정적으로 동작한다.
- 실제 체결이 없었다는 정보는 플래그로 보존할 수 있다.
- 이후 전략이 `거래 없음 구간`을 별도 필터로 활용할 수 있다.

### 12.2 체결 틱 장기 저장 위치

- 체결 틱 장기 저장은 `Supabase + 로컬 아카이브 병행`을 기본으로 한다.
- Supabase에는 최근 조회, 연구, 운영에 자주 쓰는 구간과 정규화 데이터를 저장한다.
- 장기 원본 아카이브는 로컬 서버/스토리지 쪽에 병행 저장한다.

### 이유

- 체결 틱은 장기적으로 데이터량이 크다.
- Supabase만으로 모두 적재하면 비용과 운영 부담이 빠르게 증가할 수 있다.
- 반대로 로컬에만 두면 운영 조회와 제품 기능 연결이 불편해진다.
- 따라서 `서비스 운영용 저장`과 `장기 원본 보관`을 분리하는 것이 가장 현실적이다.

### 12.3 파생 지표 계산 시점

- 파생 지표 계산은 `혼합 방식`을 기본으로 한다.
- 즉시성이 필요한 지표는 런타임 또는 수집 직후 계산한다.
- 무거운 집계, 재계산, 장기간 비교 지표는 배치 작업으로 계산한다.

### 즉시 계산 대상 예시

- warning/caution 상태 스냅샷
- 김치 프리미엄 최신값
- funding anomaly 최신 상태
- 런타임 liquidity score
- 오케스트레이터 현재 판단에 필요한 핵심 지표

### 배치 계산 대상 예시

- 장기간 EMA/RSI/MACD 재계산
- 전략 검증용 파생 피처 집계
- 연구 루프용 feature snapshot
- 장기 성과 비교용 통계

### 이유

- 모든 지표를 즉시 계산하면 수집기 부담이 커진다.
- 모든 지표를 배치로만 계산하면 오케스트레이터와 페이퍼트레이딩의 반응성이 떨어진다.
- 혼합 방식이 운영 안정성과 실시간성 사이에서 가장 균형이 좋다.

## 13. 운영상 권장 방향

- 초기에는 고빈도 전략보다 중저빈도 전략을 우선 지원한다.
- 호가 데이터는 전량 장기 저장보다 이벤트 구간 저장에 집중한다.
- 파생 데이터와 김치 프리미엄은 설명 가능한 오케스트레이션을 위해 반드시 시계열로 남긴다.
- 고용량 원본 데이터는 로컬 아카이브를 기준으로 운영하고, 제품 기능에 필요한 핵심 데이터만 Supabase에 유지한다.

## 14. 추후 조정 가능 항목

- 체결 틱의 Supabase 보관 기간
- 로컬 아카이브 압축/보관 정책
- 즉시 계산 대상 지표 범위
- 배치 계산 주기
- 저유동성 알트 결측 구간 필터 기준
