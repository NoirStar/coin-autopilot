# 13 Implementation Roadmap

## 1. 문서 목적

- 현재 PRD 묶음을 실제 구현 순서로 변환한다.
- 클로드코드가 어떤 순서로 개발하면 되는지 마일스톤 기준으로 정리한다.

## 2. 전체 원칙

- 이 문서는 `전체 프로젝트 범위`를 부정하지 않는다.
- 모든 기능은 장기적으로 구현 대상이 될 수 있다.
- 다만 구현은 `초기 구현 -> 확장 구현 -> 고도화 구현` 순으로 나눈다.
- 실전 자동매매보다 연구/페이퍼/오케스트레이션 기반을 먼저 만든다.
- UI보다 데이터와 상태 계약을 먼저 고정한다.
- 각 단계는 다음 단계를 위한 데이터와 상태를 남겨야 한다.
- 클로드코드 구현 단계에서는 `서버 구동 금지`, `외부 API 호출 금지`, `빌드 확인만 허용`을 기본 제약으로 둔다.

## 3. 전체 프로젝트 구현 단계

### Stage 1. 초기 구현

목표:

- 암호화폐 중심으로 `연구 루프 + 페이퍼 + 운영실 홈`을 실제로 돌리는 첫 구현

범위:

- Upbit + OKX 데이터 파이프라인
- 전략 카탈로그
- 자동 백테스트 연구 루프
- 기본 검증과 `paper_candidate`
- 페이퍼트레이딩
- 단순화된 오케스트레이터
- 운영실 홈 대시보드
- 핵심 알림

### Stage 2. 확장 구현

목표:

- 초기 구현에서 쌓인 데이터를 바탕으로 자동화 범위를 넓히고, 한국주식과 더 많은 전략군을 붙인다.

범위:

- 한국주식 1차 지원
- 고급 전략군 일부 확장
- 리스크/알림 고도화
- 오케스트레이터 고도화
- 더 정교한 실행 정책

### Stage 3. 고도화 구현

목표:

- 플랫폼 확장성과 멀티유저 구조, 실전 후보 체계를 완성한다.

범위:

- 멀티유저 구조
- 사용자별 전략 선택
- 사용자별 세션/알림/승인 분리
- 실전 adapter 기반 확장
- 장기 플랫폼 기능

## 4. 초기 구현 우선순위 개요

### Phase 0. 기반 계약 고정

- Schema/API contract 확정
- 상태 enum 확정
- 공통 심볼 키/UTC 기준 확정

### Phase 1. 데이터 파이프라인

- Upbit / OKX 데이터 수집
- 정규화 계층 구축
- 파생 지표 저장

### Phase 2. 전략 카탈로그 + 연구 루프

- 전략 메타데이터 구조
- 연구 run 저장
- 자동 백테스트
- 기본 검증과 후보 승격

### Phase 3. 페이퍼트레이딩 엔진

- paper session
- order/fill/position 상태
- 슬리피지/부분체결 모델
- 승인 기반 배포

### Phase 4. 오케스트레이터

- slot 상태
- 전략 후보 점수화
- 유지/교체/중지 판단
- 승인 요청 생성

### Phase 5. 운영실 대시보드

- 상단 상태 바
- 오케스트레이션 보드
- 포지션/세션 패널
- 연구 현황
- 리스크/승인/AI 액션 센터

### Phase 6. 알림 체계

- 인앱 알림
- 텔레그램
- 디스코드
- 승인/리스크 우선 알림

## 5. 확장 구현 우선순위 개요

### Phase 7. 한국주식 1차 확장

- 대형주 유니버스
- 추세/브레이크아웃 전략
- 페이퍼 검증

### Phase 8. 실행/연구 고도화

- 고급 전략군 일부 확장
- 더 정교한 실행 정책
- 리스크/알림 세분화
- 연구 루프 고도화

### Phase 9. 멀티유저 준비

- 사용자별 데이터 분리
- 사용자 전략 선택
- 사용자 알림/대시보드 설정

## 6. 초기 구현 상세 작업

### Phase 0. 계약 고정

필수 결과:

- [12_SCHEMA_AND_API_CONTRACT.md](/root/work/coin-autopilot/PRD/12_SCHEMA_AND_API_CONTRACT.md) 기준 엔티티 확정
- 상태값 enum 정의
- 기본 테이블/DTO 계약 확정

완료 기준:

- 이후 개발에서 엔티티 이름과 상태값을 다시 흔들지 않아도 된다.

### Phase 1. 데이터 파이프라인

필수 작업:

- 자산 마스터와 심볼 매핑
- OHLCV 적재
- trade tick 적재
- 파생 지표 저장
- 김치 프리미엄 저장
- UTC 기준 정렬

완료 기준:

- 연구 루프와 대시보드가 공통 데이터 계층을 읽을 수 있다.

### Phase 2. 전략 카탈로그 + 연구 루프

필수 작업:

- 전략 메타데이터 등록 구조
- 전략 추가 가능 구조
- 백테스트 run 저장
- 파라미터 세트 저장
- 승격 상태 저장

완료 기준:

- 새 전략을 메타데이터와 함께 등록할 수 있다.
- 자동 백테스트 결과가 `paper_candidate`까지 연결된다.

### Phase 3. 페이퍼트레이딩 엔진

필수 작업:

- paper session 생성/상태 전이
- paper order / fill / position
- 부분체결 모델
- 슬리피지 모델
- 승인 기반 세션 시작

완료 기준:

- 검증 통과 전략을 실제 페이퍼 세션으로 돌릴 수 있다.

### Phase 4. 오케스트레이터

필수 작업:

- asset_slot / basket_slot
- 후보 전략 랭킹
- 유지/교체/중지 판단
- approval request 생성

완료 기준:

- 시스템이 자산별 전략 추천과 교체 제안을 만들 수 있다.

### Phase 5. 운영실 대시보드

필수 작업:

- 운영실 홈
- 오케스트레이션 보드
- 포지션/세션 패널
- 연구 현황 패널
- 리스크/승인 패널

완료 기준:

- 사용자가 첫 화면에서 현재 상태와 다음 행동을 이해할 수 있다.

### Phase 6. 알림 체계

필수 작업:

- notification entity
- 인앱 알림
- 텔레그램 알림
- 디스코드 요약 알림

완료 기준:

- 승인 요청, Critical 리스크, 연구 승격 이벤트를 놓치지 않는다.

## 7. 확장 구현 상세 작업

### Phase 7. 한국주식 1차 확장

필수 작업:

- broker adapter 첫 구현
- 대형주 유니버스
- 추세/브레이크아웃 전략
- 장 시작/장 마감 이벤트 반영

완료 기준:

- 한국주식도 암호화폐와 같은 흐름으로 연구/페이퍼 검증이 가능하다.

### Phase 8. 실행/연구 고도화

필수 작업:

- 실행 엔진 세부 정책 고도화
- 연구 루프 고급 검증 일부 도입
- 후순위 전략군 일부 연구 착수
- 리스크 이벤트 세분화

완료 기준:

- 초기 구현 대비 자동화 범위와 검증 품질이 한 단계 올라간다.

### Phase 9. 멀티유저 준비

필수 작업:

- user profile
- 사용자별 전략 선택
- 사용자별 세션/승인/알림 분리
- 대시보드 개인화

완료 기준:

- 본인 전용 도구에서 멀티유저 플랫폼 구조로 넘어갈 수 있다.

## 8. 구현 순서 추천

가장 현실적인 구현 순서는 아래다.

1. 데이터 계층
2. 전략 카탈로그
3. 연구 루프
4. 페이퍼트레이딩
5. 오케스트레이터
6. 대시보드
7. 알림
8. 한국주식 확장
9. 멀티유저 준비

## 9. 첫 번째 마일스톤 정의

첫 번째로 "실제로 돌아간다"고 볼 수 있는 초기 구현 마일스톤은 아래다.

- Upbit + OKX 데이터 수집
- 전략 2~4개 등록 가능
- 자동 백테스트 run 저장
- `paper_candidate` 생성
- 승인 후 paper session 실행
- 운영실 홈에서 상태 조회 가능

## 10. 두 번째 마일스톤 정의

- 리스크 중지 규칙 작동
- 오케스트레이터 추천 작동
- 승인 요청/알림 작동
- 상위 후보 전략 자동 갱신
- 실행 엔진 상태 흐름 안정화

## 11. 세 번째 마일스톤 정의

- 한국주식 1차 지원
- 고급 전략군 일부 확장
- 사용자별 설정 구조 준비

## 12. 네 번째 마일스톤 정의

- 멀티유저 준비 레벨 도달
- 사용자별 세션/알림/승인 분리
- 플랫폼 확장 기반 확보

## 13. 의존성 주의사항

- 오케스트레이터는 연구 루프 없이 만들면 안 된다.
- 페이퍼트레이딩은 실행 엔진 상태 모델 없이 만들면 나중에 다시 뜯어야 한다.
- 대시보드는 오케스트레이터/리스크/연구 상태가 없으면 빈 껍데기가 된다.
- 멀티유저는 공용/사용자 데이터 분리 기준 없이 붙이면 큰 비용이 든다.

## 14. 전략 추가 가능 구조 확인

이 PRD 묶음은 `고정 전략 세트`를 전제로 하지 않는다.

핵심 근거:

- 전략 카탈로그 구조 존재
- 전략 메타데이터 계약 존재
- 연구 루프가 전략 템플릿과 파라미터 세트를 자동 탐색
- 오케스트레이터가 검증된 후보군을 읽어 배치

즉:

- 나중에 사용자가 전략을 추가할 수 있다.
- 새 전략은 메타데이터와 검증 경로만 맞추면 연구 루프에 편입 가능하다.
- 백테스트/페이퍼 결과에 따라 파라미터와 우선순위는 계속 바뀔 수 있다.
