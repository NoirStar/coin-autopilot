# 01 Product Vision

## 1. 문서 목적

- `coin-autopilot`이 어떤 제품이 되어야 하는지 한눈에 정리한다.
- 이후 오케스트레이터, 데이터, 백테스트, 페이퍼트레이딩, 대시보드 명세의 기준점을 제공한다.
- 대화 중 누적된 아이디어를 실행 가능한 제품 방향으로 압축한다.

## 2. 제품 한 줄 정의

- 초보 개인 운영자가 AI 오케스트레이션을 통해 암호화폐와 한국주식에 자산별 전략을 배치하고, 자동 연구 루프와 페이퍼트레이딩으로 검증해 나가는 멀티자산 자동매매 플랫폼

## 2.1 장기 플랫폼 비전

- 장기적으로는 본인만 쓰는 도구를 넘어, 다른 사용자들도 로그인해서 사용할 수 있는 전략 운용 플랫폼으로 확장한다.
- 사용자는 축적된 백테스트/페이퍼트레이딩/실전 검증 데이터를 바탕으로 성과가 좋은 전략을 선택할 수 있어야 한다.
- 사용자는 전략을 수동으로 선택해서 직접 운용할 수도 있고, 자동 오케스트레이션에 맡길 수도 있어야 한다.
- 전략 시스템은 특정 전략 몇 개를 하드코딩하는 구조가 아니라, 나중에 전략을 자유롭게 추가하고 분류하고 비교할 수 있는 확장형 구조여야 한다.
- 장기적으로는 개인 도구에서 멀티유저 플랫폼으로 성장할 수 있는 계정, 권한, 데이터 분리 구조를 고려해야 한다.

## 3. 이 제품이 아닌 것

- BTC 상태판 하나만 보여주는 대시보드
- 특정 코인 시그널을 소비하는 공개 콘텐츠 서비스
- 단일 최고 전략 하나를 고정해서 돌리는 봇
- 곧바로 실전 자동매매부터 시작하는 공격적 시스템

## 4. 핵심 문제

- 현재 구조는 BTC 레짐 중심이라 제품 전체가 특정 자산 관점에 고정되어 있다.
- 자산마다 다른 전략을 동시에 배치하고 관리하는 구조가 부족하다.
- 백테스트가 수동 기능처럼 느껴지지만, 실제 목표는 시스템이 계속 전략을 찾는 자동 연구 루프다.
- 초보 사용자가 전략 탐색, 검증, 페이퍼트레이딩, 운용을 한 흐름으로 이해하기 어렵다.
- 실전 이전에 필요한 설명, 승인, 안전장치가 충분히 구조화되어 있지 않다.

## 5. 목표 사용자

- 1차 사용자: 본인
- 사용자 특성:
  - 자동매매 초보
  - 전략을 직접 이해하고 검증하고 싶음
  - 완전 수동 운용보다 시스템 도움을 선호함
  - 실전까지 가고 싶지만, 검증과 안전장치를 먼저 원함

### 장기 확장 사용자

- 다른 개인 투자자
- 자동매매 입문자
- 전략을 직접 고르고 수동/자동 운용을 병행하고 싶은 사용자
- 축적된 전략 성과 데이터를 보고 검증된 전략을 선택하고 싶은 사용자

## 6. 제품 핵심 원칙

- 전략 중심이어야 한다.
- 오케스트레이션 중심이어야 한다.
- 검증이 운용보다 먼저여야 한다.
- 자동화는 점진적으로 확대되어야 한다.
- AI는 상시 구동보다 이벤트 기반 호출이 우선이다.
- 페이퍼와 실전의 권한 수준은 분리되어야 한다.
- "최적 전략 하나"보다 "현재 우세한 전략 후보군"을 다뤄야 한다.
- 전략 추가가 자유로운 구조여야 한다.
  - 전략은 코드 수정 없이도 등록, 분류, 비교, 비활성화, 교체가 가능한 방향을 목표로 한다.
- 수동 운용과 자동 운용이 함께 있어야 한다.
  - 사용자는 동일한 전략을 수동 모드로도, 자동 모드로도 사용할 수 있어야 한다.
- 장기적으로 멀티유저 플랫폼으로 확장 가능해야 한다.
  - 계정, 전략 선택, 성과 데이터, 알림 설정은 사용자 단위로 분리 가능해야 한다.

## 7. 현재 범위

### 포함

- 암호화폐 + 한국주식
- Upbit + OKX 중심
- Binance는 필요 시 보조 데이터 소스
- 전략 탐색
- 자유롭게 확장 가능한 전략 레지스트리 구조
- 자동 백테스트 연구 루프
- 페이퍼트레이딩
- 수동 운용 모드와 자동 운용 모드를 모두 고려한 구조
- 동적 오케스트레이션 대시보드
- AI 기반 재분석/조정 제안
- 텔레그램/디스코드 알림

### 제외

- 공개 시그널 서비스 우선 개발
- 한국주식 실전 자동매매
- 초기 단계의 완전 자율 실전매매
- 초기 단계에서의 완전한 멀티유저 서비스 운영

## 8. 핵심 사용자 흐름

1. 사용자는 첫 화면에서 운용 상태, 포지션, 시장 상황, 전략 배치 상태를 본다.
2. 시스템은 전략/지표/파라미터를 자동으로 연구하고 후보를 갱신한다.
3. 상위 후보는 페이퍼트레이딩 대상으로 승격된다.
4. 오케스트레이터는 자산별로 다른 전략을 배치하고, 성과 악화나 시장 구조 변화가 생기면 재평가한다.
5. 특정 이벤트가 발생하면 AI가 호출되어 원인 분석과 조정 제안을 제공한다.
6. 충분히 검증된 전략만 실전 후보로 승격된다.

### 장기 사용자 흐름

1. 사용자는 로그인 후 자신의 전략 목록과 운용 현황을 본다.
2. 사용자는 시스템에 등록된 전략 중 성과가 좋은 전략을 비교한다.
3. 사용자는 전략을 직접 선택해서 수동 모드로 테스트하거나, 자동 모드로 배치한다.
4. 사용자는 자신의 백테스트/페이퍼트레이딩 결과와 플랫폼 전체 또는 추천 전략의 성과 데이터를 함께 참고한다.
5. 장기적으로는 다른 사용자도 동일한 플랫폼 안에서 각자 전략과 계정을 분리해 사용한다.

## 9. 첫 화면의 성격

- 첫 화면은 `트레이딩 대시보드`이어야 한다.
- 사용자는 들어오자마자 아래를 봐야 한다.
  - 현재 잔고와 자산 상태
  - 현재 포지션
  - 오늘 손익과 누적 성과
  - 시장 상황
  - 자산별 전략 배치
  - 백테스트/연구 작업 진행 현황
  - AI 액션 필요 항목

## 10. 전략 운영 관점

- 자산마다 다른 전략을 동시에 사용할 수 있어야 한다.
- 전략은 자유롭게 추가 가능한 구조여야 한다.
- 예:
  - BTC 선물은 추세 전략
  - 특정 알트는 유동성 필터 기반 추세 전략
  - 다른 알트는 평균회귀 전략
  - 한국주식은 별도 전략군
- 전략 평가는 단순 수익률이 아니라 MDD, 승률, 리스크, 최근 적합도까지 포함해야 한다.
- 장기적으로는 전략별 성과 데이터가 누적되어, 사용자가 승률/수익률/리스크 기준으로 전략을 선택할 수 있어야 한다.
- 전략은 단순 내부 알고리즘 묶음이 아니라, 등록 가능한 제품 단위처럼 관리될 수 있어야 한다.

## 11.1 운용 모드

- 수동 모드
  - 사용자가 전략을 직접 선택하고 직접 실행 여부를 판단한다.
- 자동 모드
  - 오케스트레이터가 전략을 자산별로 배치하고 자동으로 운용한다.
- 반자동 모드
  - 시스템이 전략 후보를 추천하고 사용자가 승인한다.

## 11. AI 역할

- 상시 자동매매 엔진이 아니라 이벤트 기반 조정자 역할
- 개입 예시:
  - 성과 악화
  - 시장 구조 변화
  - 파라미터 재탐색 필요
  - 특정 시그널 발생
- 권한 원칙:
  - 페이퍼트레이딩: 자동 조정 가능
  - 실전: 승인 필요

## 12. 장기 성장 방향

- 전략 추가가 쉬운 확장형 아키텍처
- 사용자 로그인 기반 개인화
- 사용자별 전략/데이터/알림 분리
- 전략 성과 랭킹 및 추천
- 수동/반자동/자동 운용 모드 공존
- 장기적으로 다른 사용자가 사용할 수 있는 플랫폼화
