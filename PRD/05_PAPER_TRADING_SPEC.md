# 05 Paper Trading Spec

## 1. 문서 목적

- 검증된 전략을 실전 투입 전에 실제 시장 데이터 환경에서 시험하는 페이퍼트레이딩 시스템을 정의한다.
- 이 문서는 구현 지시용 PRD이며, 대화 과정은 남기지 않고 `합의된 정책`만 기록한다.

## 2. 한 줄 정의

- 페이퍼트레이딩은 실자산 없이 실제 시장 데이터와 유사한 체결/비용/지연 조건을 반영해 전략의 실현 가능성과 운영 안정성을 검증하는 `실전 직전 검증 환경`이다.

## 3. 범위

### 포함

- 페이퍼 세션 구조
- 전략 배포 방식
- 주문 시뮬레이션
- 슬리피지/부분체결/실패 처리
- 오케스트레이터 연계
- AI 자동 조정 권한
- 성과 측정과 통과 기준
- 알림 및 승인 흐름

### 제외

- 실전 주문 API 집행
- 실계좌 잔고 동기화
- 실전 체결 복구 로직 상세
- 공개 사용자용 플랫폼 운영 정책

## 4. 역할

페이퍼트레이딩은 단순히 가짜 돈으로 차트를 돌리는 기능이 아니다.

- 연구 루프가 만든 후보 전략을 실제 시장 시간축 위에서 검증한다.
- 백테스트와 실제 운영 사이의 차이를 발견한다.
- 슬리피지, 부분체결, 레이턴시, 데이터 결측, 주문 실패를 드러낸다.
- 오케스트레이터가 전략 유지/교체/중지 판단을 내릴 수 있는 운영 데이터를 만든다.

## 5. 핵심 원칙

- 백테스트 검증 데이터 없는 전략은 페이퍼 배포 금지
- 초기 V2는 `추천 + 승인` 기반 배포
- 충분한 검증 데이터가 쌓인 뒤에만 페이퍼 자동 교체 확대
- 기대값, 총수익, MDD를 승률보다 우선 평가
- 페이퍼 결과는 실전 후보 승격의 필수 근거다.
- 페이퍼 환경은 낙관적인 체결 가정을 피하고 보수적으로 모델링한다.

## 6. V2 적용 범위

### 자산군

- 암호화폐
- 한국주식

### 채널

- Upbit 현물
- OKX 선물

### 운용 범위

- 실전 자동매매는 직접 포함하지 않음

## 7. 페이퍼 세션 구조

### 7.1 세션 단위

- 하나의 페이퍼 세션은 다음 조합으로 정의한다.
  - 전략
  - 자산 또는 자산 바스켓
  - 운용 모드
  - 시작 자본
  - 리스크 정책
  - 비용 모델

### 7.2 세션 상태

- `draft`
- `approval_pending`
- `running`
- `paused`
- `stopped`
- `completed`
- `failed`

### 7.3 세션 메타데이터

- session_id
- strategy_id
- asset_slot 또는 basket_slot
- deployment_mode
- initial_capital
- current_equity
- current_drawdown
- approval_status
- created_at
- started_at
- ended_at

### 7.4 V2 기본 세션 자본

V2에서는 전략 비교와 운영 기준 통일을 위해 자산군별 기본 가상 자본을 고정한다.

#### OKX 선물

- 기본 세션 자본: `10,000 USDT`

#### Upbit 현물

- 기본 세션 자본: `10,000,000 KRW`

#### 한국주식

- 기본 세션 자본: `10,000,000 KRW`

### 이유

- 전략 간 비교 기준을 통일할 수 있다.
- 최소 주문 금액보다 충분히 크다.
- 수수료, 슬리피지, 부분체결의 영향을 관찰하기에 너무 작지 않다.
- 과도하게 큰 자본으로 비현실적인 체결을 가정하는 것도 피할 수 있다.

### 적용 원칙

- 단일 자산 전략은 해당 세션 자본을 기준으로 운용한다.
- 바스켓 전략은 세션 자본을 전략 비중 규칙에 따라 분할한다.
- 사용자는 추후 사용자 설정으로 기본 자본을 조정할 수 있지만, V2 기본 비교 기준은 위 값을 사용한다.

## 8. 페이퍼 배포 정책

### 8.1 배포 전제

- 전략은 `paper_candidate` 상태여야 한다.
- 백테스트와 검증 기준을 통과해야 한다.
- 필요한 데이터 소스가 안정적이어야 한다.

### 8.2 V2 기본 배포 방식

- 초기 V2는 `추천 + 승인` 방식
- 즉:
  - 오케스트레이터가 배포 후보를 제안
  - 사용자가 승인
  - 승인 후 세션 생성 및 실행

### 8.3 추후 자동 확대

- 충분한 페이퍼 검증과 운영 안정성이 확보되면
  - 동일 전략군 내 교체
  - 파라미터 조정
  - 비중 축소
  를 자동 허용할 수 있다.

## 9. 주문 시뮬레이션 모델

### 9.1 기본 원칙

- 백테스트보다 더 보수적인 체결 가정을 사용한다.
- 시장가/지정가/최유리/IOC/FOK 등의 차이를 반영한다.
- 수수료와 슬리피지를 반드시 반영한다.

### 9.2 주문 타입별 기본 처리

#### 시장가

- 즉시 체결 시도
- 슬리피지 반영
- 유동성 부족 시 부분체결 가능

#### 지정가

- 가격 도달 여부 확인 후 체결
- 체결 대기 시간 동안 미체결 상태 유지 가능
- 일정 조건에서 취소 또는 재주문

#### IOC/FOK/Post-only 성격 주문

- 거래소별 모델 차이를 반영하되, 초기 V2는 단순화 모델로 시작
- 주문 정책 차이는 execution policy 메타데이터로 연결

## 10. 슬리피지 모델

### 10.1 기본 방향

- V2는 완전 정밀한 호가 리플레이보다 `보수적 단순 모델`을 기본으로 한다.
- 다만 이벤트 전략과 저유동성 자산은 가중 슬리피지를 적용한다.

### 10.2 기본 정책

- 저빈도 전략:
  - 고정 bp 슬리피지 + 변동성 구간 가산
- 중빈도 전략:
  - 고정 bp + 거래대금/유동성 기반 가산
- 고빈도/호가 전략:
  - 후순위
  - 데이터 파이프라인 성숙 전까지 적극 배포하지 않음

### 10.3 가중 조건

- warning/caution 상태
- 변동성 급등
- 9시 이벤트 구간
- 청산 이벤트 구간
- 거래대금 급감

### 10.4 V2 기본 슬리피지 값

아래 수치는 거래소 공식 주문 구조, 수수료 체계, 최소 주문/호가 구조, 자산군별 유동성 특성을 바탕으로 설정한 `V2 기본 운영값`이다.

#### OKX BTC/ETH 선물

- 지정가 체결: `0.5 bp`
- 시장가/테이커 체결: `2 bp`
- 이벤트 구간 가산:
  - 고변동 구간 `+5 bp`
  - 청산 급증/펀딩 이벤트/급격한 체결 왜곡 구간 `+10 bp`

#### Upbit 알트 현물

- 지정가 체결: `3 bp`
- 시장가/최유리 체결: `12 bp`
- 이벤트 구간 가산:
  - 일반 급등락 구간 `+20 bp`
  - 09:00 KST 이벤트, warning/caution, 유동성 급감 구간 `+35 bp`

#### 한국주식 1차 대형주

- 지정가 체결: `1 bp`
- 시장가/공격적 체결: `5 bp`
- 이벤트 구간 가산:
  - 장 시작/장 마감 급변 구간 `+10 bp`
  - 급격한 변동성 확대 구간 `+15 bp`

### 적용 원칙

- 기본값은 `시장가 기준 보수적 시뮬레이션`에 사용한다.
- 실제 전략별 execution policy가 지정가 중심이면 해당 전략은 지정가 기준값을 우선 사용한다.
- 후속 데이터 축적 후 자산별/전략별 override가 가능하다.

## 11. 부분체결 정책

### 11.1 기본 원칙

- V2 페이퍼트레이딩은 `부분체결 가능`을 기본 가정으로 한다.
- 전량 체결만 가정하는 낙관적 모델은 사용하지 않는다.

### 11.2 처리 방식

- 주문량 일부만 체결될 수 있다.
- 미체결 잔량은 별도 상태로 남긴다.
- 전략은 미체결 잔량을 포함한 현재 포지션 상태를 기준으로 다시 판단한다.

### 11.3 세션 영향

- 부분체결이 반복되면 execution quality 경고를 발생시킨다.
- 특정 전략이 지속적으로 체결 품질이 나쁘면 오케스트레이터가 감점한다.

### 11.4 V2 기본 부분체결 모델 강도

- 기본 모델 강도는 `중간보다 약간 보수적`으로 둔다.

#### 공통 원칙

- 주문이 호가에 닿았다는 이유만으로 전량 체결로 간주하지 않는다.
- 지정가 주문은 `touch-fill` 낙관 가정을 사용하지 않는다.
- 거래량과 유동성 대비 주문 크기를 기준으로 즉시 체결량을 제한한다.

#### OKX BTC/ETH 선물

- 추정 즉시 유동성 대비 주문 비중이 `10% 이하`면 전량 체결 가능
- `10% 초과 ~ 30% 이하`면 `70~90%` 즉시 체결, 잔량은 다음 틱/주기로 이월
- `30% 초과`면 `40~70%`만 즉시 체결, 잔량은 분할 체결 또는 재주문 처리

#### Upbit 알트 현물

- 추정 즉시 유동성 대비 주문 비중이 `5% 이하`면 전량 체결 가능
- `5% 초과 ~ 20% 이하`면 `60~85%` 즉시 체결
- `20% 초과`면 `30~60%`만 즉시 체결, 잔량은 후속 체결 또는 취소/재시도

#### 한국주식 1차 대형주

- 추정 즉시 유동성 대비 주문 비중이 `10% 이하`면 전량 체결 가능
- `10% 초과 ~ 25% 이하`면 `70~90%` 즉시 체결
- `25% 초과`면 `40~70%` 즉시 체결

### 지정가 주문 원칙

- 체결 조건은 단순 가격 도달이 아니라 `거래 체결 또는 충분한 호가 소모`를 기준으로 본다.
- 즉, 단순 터치만으로는 체결 처리하지 않는다.

## 12. 실패 및 예외 처리

페이퍼트레이딩은 아래 상황을 모델링해야 한다.

- 데이터 지연
- 데이터 결측
- 주문 실패
- 부분체결
- 비정상 슬리피지 확대
- 시스템 일시 중지

### 기본 정책

- 오류는 세션 로그에 기록한다.
- 반복 오류 시 세션을 자동 `paused` 상태로 전환할 수 있다.
- 심각한 오류는 승인 필요 알림 또는 AI 리뷰 요청을 만든다.

## 13. 리스크 관리

### 13.1 기본 정책

- 일 손실 한도
- 전략별 최대 MDD 한도
- 자산별 최대 노출 한도
- 레버리지 한도
- 비정상 이벤트 시 자동 중지

### 13.2 서킷 브레이커

- 급격한 변동성 확대
- 데이터 이상
- 주문 실패 반복
- 경고 이벤트 급증

위 조건에서 세션을 일시 중지하거나 비중을 축소할 수 있다.

## 15. 페이퍼 통과 기준

연구 루프와 오케스트레이터에서 합의한 V2 기본 기준을 따른다.

- 최소 30일 이상 페이퍼트레이딩
- 최소 50회 이상 청산 완료 거래
- 비용 반영 후 기대값 양수
- 총손익 양수
- 전략별 허용 MDD 이하
- 반복적인 운영 오류 없음
- 백테스트 대비 구조적 붕괴 없음

### 기본 MDD 가이드

- 저위험 전략: 10% 이하
- 중위험 전략: 15% 이하
- 고위험 전략: 20% 이하

## 16. 오케스트레이터 연계

- 오케스트레이터는 `paper_running` 세션을 실시간 감시한다.
- 성과 악화, 시장 구조 변화, 리스크 이벤트가 생기면 재평가한다.
- 초기 V2에서는 전략 교체/배포 시 사용자 승인 필요
- 충분한 데이터가 쌓이면 동일 전략군 내 자동 조정 확대 가능

## 17. AI 권한

### V2 기본 원칙

- 페이퍼트레이딩에서는 AI가 자동 조정에 개입할 수 있다.
- 다만 초기 V2 기본 배포/교체는 사용자 승인 이후 반영한다.
- AI는 원인 분석, 파라미터 조정 제안, 교체 추천, 중지 추천을 할 수 있다.

### AI 자동 조정 허용 범위

- 동일 전략군 내 파라미터 미세 조정
- 페이퍼 비중 축소
- 리스크 이벤트 대응 중지

### AI 자동 조정 금지 범위

- 실전 승격 자동 승인
- 검증 없는 신규 전략 배포
- 근거 없는 공격적 비중 확대

## 18. 알림 및 승인 흐름

### 사용자에게 알려야 하는 항목

- 신규 페이퍼 배포 승인 요청
- 페이퍼 성과 악화
- MDD 한도 근접
- 반복 주문 실패
- AI 재분석 결과
- 실전 후보 승격 제안

### 채널

- 대시보드

- 디스코드
