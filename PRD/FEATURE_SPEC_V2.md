# Coin Autopilot V2 기능 명세서

상태: 초안 작성 중  
작성 방식: 대화 기반으로 단계별 업데이트  
기존 구현 분석 문서: `CURRENT_IMPLEMENTATION_AUDIT.md` 유지

## 1. 제품 정의

### 1.1 한 줄 정의

- 초보 개인 운영자가 AI 오케스트레이션을 통해 암호화폐와 한국주식에 자산별 전략을 배치하고, 검증부터 페이퍼트레이딩까지 이어갈 수 있게 돕는 멀티자산 자동매매 플랫폼

### 1.2 해결하려는 문제

- 현재 구조는 특정 관점(BTC 레짐)에 너무 고정되어 있어 시장이 바뀌면 제품 중심도 같이 흔들린다.
- 사용자는 "지금 어떤 전략이 가장 유리한가"를 보고 싶은데, 현재 UI는 "특정 지표를 보여주는 대시보드"에 가깝다.
- 실제로는 자산마다 유리한 전략이 다를 수 있는데, 현재 구조는 여러 자산에 서로 다른 전략을 배치하고 오케스트레이션하는 모델을 담지 못한다.
- 초보 사용자가 전략 탐색, 백테스트, 페이퍼트레이딩, 실전 운용을 한 흐름으로 이어가기가 어렵다.
- 백테스트가 현재는 사용자가 직접 실행하는 기능처럼 느껴지기 쉬운데, 궁극적으로 필요한 것은 시스템이 자동으로 전략/지표/파라미터를 탐색하고 계속 더 좋은 조합을 찾는 연구 루프다.
- 앞으로는 코인, 주식, 선물, 현물, 한국주식 등 다양한 시장을 지원하고 싶기 때문에 특정 거래소/특정 자산 전용 구조로는 확장성이 부족하다.
- 장기적으로는 AI가 분석, 파라미터 탐색, 운용 보조까지 개입하는 형태를 염두에 두고 있기 때문에, 처음부터 전략 중심 플랫폼 구조가 필요하다.

### 1.3 목표 사용자

- 1차 사용자: 본인
- 사용자 특성:
  - 자동매매 초보
  - 직접 전략을 검증하고 싶음
  - 실전 운용까지 염두에 두고 있음
  - 숫자와 전략을 완전히 수동으로 관리하기보다는 시스템과 AI의 도움을 받고 싶음
- 제품은 초보자가 이해 가능한 설명, 안전장치, 단계별 진입 흐름을 기본 전제로 설계해야 한다.

### 1.4 사용 시나리오

- 사용자는 첫 화면에서 현재 자동운용 상태를 한눈에 확인한다.
  - 현재 보유 포지션
  - 잔고
  - 누적 성과
  - 승률
  - 현재 시장 상황
  - 백테스트/분석 작업 진행상황
- 사용자는 "하나의 최고 전략"이 아니라, 자산별로 어떤 전략이 현재 적용되고 있는지 본다.
  - 예: 특정 코인은 추세 전략
  - 예: 다른 코인은 평균회귀 전략
  - 예: 특정 한국주식은 브레이크아웃 전략
- 사용자는 특정 자산군에서 쓸 수 있는 전략 후보를 탐색한다.
- 시스템은 여러 전략, 지표, 파라미터 조합을 자동으로 백테스트한다.
- 시스템은 여러 스레드/작업 단위로 장시간 분석을 돌리며 더 나은 파라미터나 전략 조합을 찾는다.
- 사용자는 매번 수동 실행하기보다, 탐색 범위와 정책을 설정하고 결과를 확인한다.
- 사용자는 페이퍼트레이딩으로 전략을 실제 시장 조건에서 검증한다.
- 시스템은 상위 백테스트 결과를 바탕으로 페이퍼트레이딩 후보를 자동으로 올릴 수 있어야 한다.
- 사용자는 특정 시그널이나 이상 상황이 발생했을 때만 AI를 호출해 원인 분석, 수치 조정 제안, 전략 조합 재평가를 받는다.
- 사용자는 신뢰가 쌓인 전략만 실전 자동매매로 전환한다.
- 사용자는 대시보드에서 특정 자산이 아니라 "지금 어떤 자산에 어떤 전략이 배치되어 있고, 왜 그렇게 판단했는지"를 중심으로 본다.
- 사용자는 주요 변화와 결과를 텔레그램/디스코드로 받아본다.

## 2. 제품 원칙

- 오케스트레이션 중심이어야 한다.
  - 시스템은 단일 전략 실행기가 아니라, 자산별로 적합한 전략을 배치하고 조정하는 오케스트레이터여야 한다.
- 전략 중심이어야 한다.
  - 대시보드는 BTC 같은 특정 자산 중심이 아니라, 현재 유효한 전략과 운용 상태 중심으로 구성한다.
- 초보 친화적이어야 한다.
  - 각 전략, 위험도, 파라미터, 시그널의 의미를 이해 가능한 언어로 설명해야 한다.
- 검증이 운용보다 먼저여야 한다.
  - 전략 생성 -> 백테스트 -> 페이퍼트레이딩 -> 실전 전환의 흐름이 기본이다.
- 연구 루프는 자동화되어야 한다.
  - 백테스트는 수동 버튼 실행 기능이 아니라, 전략/지표/파라미터를 자동 탐색하는 백그라운드 연구 시스템이어야 한다.
- 자동화는 점진적으로 확대되어야 한다.
  - 처음부터 완전 자율 운용보다, 추천/보조/승인 기반 자동화에서 시작해 신뢰를 쌓는다.
- AI는 마법이 아니라 운영 보조자여야 한다.
  - AI는 전략 발견, 파라미터 조정, 이상 감지, 재평가에 관여하되, 어느 단계까지 자동 결정권을 갖는지는 명확히 구분한다.
- AI는 상시 구동이 아니라 이벤트 기반 호출을 우선한다.
  - 비용을 고려해 항상 켜두는 구조보다, 특정 시그널/이상상황/재평가 요청 시 한 번씩 호출되는 구조를 기본으로 한다.
- 페이퍼와 실전의 AI 권한은 달라야 한다.
  - 페이퍼트레이딩에서는 AI 자동 조정을 허용할 수 있지만, 실전에서는 승인 기반이어야 한다.
- 멀티자산 확장이 가능해야 한다.
  - 특정 거래소나 특정 자산을 전제로 한 화면/데이터 모델은 피한다.
- "최적의 전략"은 고정된 하나가 아니라 후보군으로 다뤄야 한다.
  - 시장, 자산, 기간, 리스크 설정에 따라 우수 전략은 달라질 수 있으므로, 시스템은 단일 정답보다 현재 우세한 전략 후보를 탐색/정렬/추천해야 한다.

## 3. 범위

### 3.1 포함 범위

- V2 1차 지원 자산군:
  - 암호화폐
  - 한국주식
- V2 1차 주요 거래 채널:
  - Upbit
  - OKX
- V2 1차 한국주식 범위:
  - 백테스트
  - 페이퍼트레이딩
- Binance 및 기타 거래소 데이터는 필요 시 보조 데이터 소스로 확장 가능해야 한다.
- 전략 레지스트리
  - 자산군별 전략을 등록, 분류, 비교할 수 있어야 한다.
- 오케스트레이션 레이어
  - 자산별로 어떤 전략을 배치할지 결정하고 상태를 관리해야 한다.
- 전략 검증 워크플로우
  - 백테스트, 파라미터 탐색, 성과 비교를 한 흐름으로 제공해야 한다.
- 자동 연구 루프
  - 시스템이 주기적 또는 이벤트 기반으로 전략/지표/파라미터 조합을 계속 탐색해야 한다.
- 페이퍼트레이딩
  - 백테스트를 통과한 전략을 실제 시장 데이터로 검증할 수 있어야 한다.
- 실전 자동매매는 V2 1차 핵심 범위는 아니다.
  - 다만 향후 승격을 고려한 데이터 구조와 승인 흐름은 준비해야 한다.
- 동적 대시보드
  - 현재 우세 전략, 운용 상태, 리스크, 자산별 상태를 유동적으로 보여줘야 한다.
- AI 운용 보조
  - 이상 감지, 재분석, 수치 조정 제안, 전략 재평가를 지원해야 한다.
- 외부 알림 연동
  - 텔레그램
  - 디스코드

### 3.2 제외 범위

- 공개용 시그널 콘텐츠 서비스는 현재 우선순위가 아니다.
- 특정 자산 하나만을 위한 하드코딩된 UX는 지양한다.
- 처음부터 모든 자산군을 완전 지원하는 것은 목표가 아니다.
- AI의 완전 자율 실전매매는 초기 단계에서 바로 기본값으로 두지 않는다.
- V2 1차에서는 전략 탐색, 백테스트, 페이퍼트레이딩이 우선이며 실전 자동매매는 후순위다.
- 한국주식 실전 자동매매는 V2 1차 범위에서 제외한다.

## 4. 핵심 사용자 흐름

### 4.1 첫 진입

- 사용자는 첫 화면에서 오케스트레이션 대시보드를 본다.
- 사용자는 현재 자산별 전략 배치 상태와 포지션 상태를 바로 확인한다.
- 초기 설정 단계에서는 거래소/브로커 연결과 기본 리스크 설정을 완료한다.

### 4.2 전략 탐색 / 선택

- 사용자는 자산군별 전략 후보를 탐색한다.
- 사용자는 성과, 리스크, 최근 시장 적합도 기준으로 전략 후보를 비교한다.
- 사용자는 단일 전략 또는 전략 묶음을 선택한다.
- 사용자는 자산마다 다른 전략을 배치할 수 있어야 한다.

### 4.3 검증

- 시스템은 선택한 전략군을 자동으로 백테스트한다.
- 시스템은 여러 지표 조합과 파라미터 조합을 자동 탐색한다.
- 시스템은 장시간 반복 분석 작업을 돌려 더 나은 설정을 찾는다.
- 사용자는 탐색 범위, 우선순위, 제약 조건을 설정하고 결과를 검토한다.
- 사용자는 AI에게 성과 부진 원인과 조정 방향을 묻는다.

### 4.4 운용

- 사용자는 전략을 페이퍼트레이딩으로 배포한다.
- 사용자는 오케스트레이터가 자산별로 어떤 전략을 적용하고 있는지 본다.
- 사용자는 특정 이벤트가 발생했을 때 AI가 전략 유지/교체/수치 조정 제안을 하는 흐름을 사용한다.
- 시스템은 상위 백테스트 결과를 페이퍼트레이딩 후보로 자동 반영할 수 있어야 한다.
- 사용자는 일정 기준 통과 시 실전 자동매매 후보로 승격한다.
- 사용자는 운용 중에도 전략을 중지, 교체, 재설정할 수 있다.

### 4.5 모니터링

- 사용자는 대시보드에서 현재 우세 전략, 현재 보유 포지션, 계좌 상태, 위험도, 이상 이벤트를 본다.
- 사용자는 자산별 전략 배치 현황을 한눈에 볼 수 있어야 한다.
- 사용자는 백테스트가 몇 개 돌고 있는지, 어떤 작업이 진행 중인지, 어떤 조합이 좋은 결과를 내는지 바로 확인할 수 있어야 한다.
- 사용자는 지표별/전략별 백테스트 결과 비교를 한 화면 또는 바로 이동 가능한 구조로 볼 수 있어야 한다.
- 특정 시그널, 손실 확대, 성과 저하, 체결 이상 시 알림을 받는다.
- 필요 시 AI가 개입해 원인 분석과 조정 제안을 제공한다.

## 5. 기능 명세

### 5.1 대시보드

- 단일 최고 전략 화면이 아니라 오케스트레이션 대시보드여야 한다.
- 자산 중심과 전략 중심 정보를 같이 보여주되, 핵심은 "어떤 자산에 어떤 전략이 배치됐는가"여야 한다.
- 동시에 "내 운용 상태"가 첫 화면에서 바로 보여야 한다.
- 현재 활성 전략 목록과 상태를 보여줘야 한다.
- 현재 자동운용 포지션을 보여줘야 한다.
- 현재 계좌 잔고와 평가손익, 누적 수익률, 승률 등 핵심 성과를 보여줘야 한다.
- 현재 시장 상황을 요약해서 보여줘야 한다.
  - 예: 시장 위험도, 자산군별 분위기, 변동성, 추세 상태
- 전략별 최근 성과, 드로우다운, 승률, 최근 시그널, 현재 적용 자산군을 보여줘야 한다.
- 백테스트/분석 작업 현황을 대시보드에서 바로 볼 수 있어야 한다.
  - 실행 중인 작업 수
  - 최근 완료된 작업
  - 가장 좋은 결과를 내는 조합
- 자동 연구 루프 상태를 보여줘야 한다.
  - 현재 어떤 전략군을 탐색 중인지
  - 어떤 지표 조합을 시험 중인지
  - 현재 가장 유리한 후보가 무엇인지
- 자산별 전략 배치 보드가 필요하다.
  - 자산명
  - 현재 적용 전략
  - 상태
  - 최근 판단 이유
  - AI 개입 여부
- 사용자는 "지금 뭘 해야 하는지"를 알 수 있어야 한다.
  - 예: 페이퍼 승격 추천, 성과 악화 경고, 파라미터 재탐색 필요
- 시장 상황 변화에 따라 대시보드 상단 핵심 카드가 유동적으로 바뀔 수 있어야 한다.
- "왜 이 전략이 지금 추천되는지" 설명이 있어야 한다.

  - 현재 주의할 이벤트

  - 어떤 자산에 어떤 전략이 적용 중인지
  - 왜 그렇게 판단했는지
  - AI 개입 여부
- 4순위: 전략 상태
  - 현재 활성 전략

  - 추천 전략 후보

위 신호들은 대시보드에 직접 노출하고, 동시에 오케스트레이터의 내부 판단 근거로도 사용한다.

### 5.2 오케스트레이터 / 전략 배치 엔진

- 시스템은 단순 전략 실행기가 아니라 전략 배치 엔진이어야 한다.
- 자산별로 어떤 전략을 적용할지 관리해야 한다.
- 같은 시점에도 자산마다 다른 전략이 적용될 수 있어야 한다.
- 오케스트레이터는 다음 정보를 관리해야 한다.
  - 자산
  - 후보 전략군
  - 현재 선택 전략
  - 선택 이유
  - 최근 성과
  - 현재 상태
- AI는 이 오케스트레이터의 판단 보조 레이어로 작동해야 한다.
- AI는 상시 실행이 아니라 특정 이벤트에서만 호출한다.
  - 예: 성과 급락
  - 예: 이상 시그널
  - 예: 페이퍼 중 재조정 필요
  - 예: 전략 교체 후보 발생
- 전략 재배치의 1차 트리거:
  - 성과 악화
  - 시장 구조 변화

#### 시장 구조 변화 예시

- 추세장에서 횡보장으로 전환
- 변동성 급등 또는 급감
- 현물-선물 괴리 확대
- 롱/숏 비율 급변
- 미결제약정 급변
- 자산군 리더십 변화

### 5.3 전략

- 전략은 자산군, 시장, 방향성, 리스크 성격으로 분류되어야 한다.
- 전략별 파라미터를 수정할 수 있어야 한다.
- 전략별 사용 가능 모드(백테스트/페이퍼/실전)를 구분해야 한다.
- 전략 단독 운용뿐 아니라 전략 포트폴리오 운용도 고려해야 한다.

### 5.4 백테스트

- 단일 실행보다 자동 반복 탐색과 비교가 더 중요하다.
- 여러 파라미터 탐색 작업을 큐/작업 단위로 돌릴 수 있어야 한다.
- 장시간 분석을 위한 비동기 작업 관리가 필요하다.
- 여러 전략, 지표, 파라미터 조합을 자동으로 생성해 평가할 수 있어야 한다.
- 사용자가 직접 매번 실행하지 않아도 백그라운드에서 지속적으로 돌아갈 수 있어야 한다.
- 사용자는 백테스트가 "지금 어떻게 돌아가고 있는지" 볼 수 있어야 한다.
  - 실행 중
  - 대기 중
  - 완료
  - 실패
- 사용자는 작업 단위로 결과를 추적할 수 있어야 한다.
  - 어떤 전략인지
  - 어떤 파라미터 범위인지
  - 어떤 시장/기간인지
  - 현재 진행률이 어떤지
- 사용자는 지표별 결과를 비교할 수 있어야 한다.
  - 수익률
  - Sharpe
  - MDD
  - 승률
  - 거래 수
  - 보유 기간
- 사용자는 전략별 비교뿐 아니라 "지표 기준 비교"도 할 수 있어야 한다.
  - 예: Sharpe 상위 조합
  - 예: MDD 가장 낮은 조합
  - 예: 승률이 높은 조합
- 사용자는 백테스트 결과를 그대로 페이퍼 후보로 보낼 수 있어야 한다.
- 시스템은 상위 결과를 자동으로 페이퍼 후보로 승격할 수 있어야 한다.
- 향후 AI가 결과를 읽고 파라미터 조정 제안을 할 수 있어야 한다.

#### 백테스트 1차 화면 요소

- 실행 패널
  - 전략군 선택
  - 시장 선택
  - 기간 선택
  - 지표 후보 선택
  - 파라미터 범위 설정
  - 자동 탐색 on/off
- 작업 모니터
  - 현재 실행 중인 작업 리스트
  - 진행률
  - 예상 남은 작업 수
- 결과 테이블
  - 전략/파라미터 조합별 핵심 지표
  - 지표 조합별 핵심 지표
- 결과 상세
  - 에퀴티 커브
  - 거래 내역
  - 성과 요약
- 비교 보기
  - 지표별 정렬
  - 상위 결과 묶음 보기
- AI 개입 방식
  - 기본은 비활성
  - 사용자가 요청하거나 특정 이벤트 조건이 맞을 때만 백테스트 결과를 읽고 파라미터 조정 제안을 수행
- AI 제안 결과는 작업 로그로 남겨야 한다.

#### 자동 연구 루프

- 시스템은 다음 단위들을 자동으로 조합해 실험할 수 있어야 한다.
  - 전략
  - 지표 세트
  - 파라미터 범위
  - 시장
  - 기간
- 실험 결과는 점수화되어 랭킹되어야 한다.
- 상위 결과는 페이퍼트레이딩 후보 큐로 넘어갈 수 있어야 한다.
- 사용자는 자동 탐색 정책을 조절할 수 있어야 한다.
  - 실행 주기
  - 최대 동시 작업 수
  - 평가 기준
  - 승격 기준

#### V2 1차 백테스트 범위

- 암호화폐:
  - Upbit
  - OKX
- 한국주식:
  - 백테스트 지원
- 실전 자동매매 승격은 후순위이며, 우선은 전략 검증과 페이퍼트레이딩 연결에 집중한다.

### 5.5 가상매매

- 백테스트 이후 반드시 거치는 실험 단계여야 한다.
- 전략별/세션별 포지션, 체결, 성과를 실제로 확인할 수 있어야 한다.
- 단순 세션 생성보다 운용 관찰과 비교가 중요하다.
- 특정 시그널 발생 시 AI가 개입해 조정 제안을 줄 수 있어야 한다.
- 페이퍼트레이딩에서는 특정 조건에서 AI 자동 조정을 허용할 수 있어야 한다.
  - 예: 드로우다운 악화
  - 예: 시장 구조 변화
  - 예: 시그널 품질 악화
- 다만 자동 조정 이력은 반드시 기록되어야 한다.
- 자동 연구 루프에서 올라온 상위 전략 후보가 실제로 어떻게 페이퍼에서 동작하는지 추적할 수 있어야 한다.

### 5.6 실전매매

- 실전 운용은 점진적으로 활성화되어야 한다.
- 전략별 승격 기준이 있어야 한다.
- 자동 정지, 손실 제한, 수동 승인 등 안전장치가 필요하다.
- AI가 실전 단계에서 어떤 수준까지 개입할 수 있는지는 별도 정책으로 정의해야 한다.
- 초기 정책:
  - AI는 실전에서 자동 변경 권한이 없다.
  - AI는 승인 요청과 조정 제안만 할 수 있다.

### 5.7 포트폴리오 / 자산

- 자산 화면은 실잔고, 운용 포지션, 전략별 배분 상태를 명확히 구분해야 한다.
- 자산군이 늘어나도 같은 정보 구조로 확장 가능해야 한다.
- "전체 자산 관점"과 "전략별 자산 관점"을 모두 보여줄 수 있어야 한다.

## 8. 성공 기준

- 사용자가 특정 전략이 아니라 전략 후보군을 비교해서 선택할 수 있다.
- 사용자가 백테스트 -> 페이퍼 -> 실전으로 자연스럽게 이동할 수 있다.
- 대시보드가 특정 자산 하드코딩 없이 현재 우세 전략과 자산별 전략 배치를 중심으로 동작한다.
- 사용자가 첫 화면만 봐도 현재 포지션, 잔고, 성과, 시장상황, 백테스트 진행상황을 파악할 수 있다.
- 사용자가 백테스트 결과를 전략별뿐 아니라 지표별로 비교할 수 있다.
- 사용자가 수동으로 계속 돌리지 않아도 시스템이 자동으로 더 나은 전략/지표/파라미터 조합을 탐색한다.
- 상위 백테스트 결과가 자연스럽게 페이퍼트레이딩으로 이어진다.
- 사용자가 자산별로 서로 다른 전략이 동시에 운용 중인 상태를 이해할 수 있다.
- 초보 사용자가 현재 상태와 다음 행동을 이해할 수 있다.
- AI 개입이 로그와 설명을 남기며 통제 가능하게 동작한다.
