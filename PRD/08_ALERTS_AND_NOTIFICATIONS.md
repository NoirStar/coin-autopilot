# 08 Alerts And Notifications

## 1. 문서 목적

- 시스템 내에서 발생하는 전략, 리스크, 승인, 연구, 실행 이벤트를 사용자에게 어떤 우선순위와 어떤 채널로 전달할지 정의한다.
- 이 문서는 구현 지시용 PRD이며, 대화 과정은 남기지 않고 `합의된 정책`만 기록한다.

## 2. 한 줄 정의

- 알림 시스템은 단순 로그 전달이 아니라, 사용자가 놓치면 안 되는 이벤트를 적절한 시점과 채널로 전달해 `빠른 판단과 승인`을 돕는 행동 유도 시스템이다.

## 3. 핵심 원칙

- 모든 이벤트를 즉시 보내지 않는다.
- 중요도와 즉시성에 따라 채널을 다르게 쓴다.
- 승인이 필요한 이벤트는 가장 먼저 보여야 한다.
- 리스크와 실행 실패는 전략 성과보다 우선적으로 알려야 한다.
- 대시보드와 외부 알림은 동일 이벤트 체계를 공유해야 한다.
- 초보 사용자도 바로 이해할 수 있는 문구를 사용해야 한다.

## 4. 채널 구조

### 4.1 대시보드 인앱 알림

- 기본 채널
- 모든 이벤트는 우선 대시보드에서 조회 가능해야 한다.
- 이력 조회와 상세 확인에 적합하다.

### 4.2 텔레그램

- 즉시성 높은 운영 알림 채널
- 승인 요청, 리스크 경고, 세션 중지, 전략 교체 제안에 적합하다.

### 4.3 디스코드

- 요약형 운영 채널
- 연구 루프 진행, 상위 후보 갱신, 세션 상태 변화, 성과 요약에 적합하다.

## 5. 알림 우선순위 체계

### 5.1 Critical

- 즉시 확인이 필요한 이벤트
- 텔레그램 + 대시보드 필수
- 필요 시 디스코드에도 동시 전송

예시:

- 일 손실 한도 초과
- 세션 자동 중지
- 주문 실패 반복
- 승인 필요 전략 교체
- 실전 후보 승격 승인 요청
- 서킷 브레이커 발동

### 5.2 Warning

- 빠른 확인이 필요한 이벤트
- 대시보드 필수
- 텔레그램 또는 디스코드 선택 전송

예시:

- MDD 한도 근접
- 부분체결 빈도 증가
- 페이퍼 성과 악화
- warning/caution 발생
- 실행 직전 재검증 실패

### 5.3 Info

- 참고용 또는 요약용 이벤트
- 대시보드 기본
- 디스코드 요약 전송 적합

예시:

- 연구 루프 완료
- 상위 후보 갱신
- 페이퍼 세션 시작/종료
- AI 리뷰 결과 요약

## 6. 이벤트 분류

### 6.1 승인 이벤트

- 신규 페이퍼 세션 배포 승인 요청
- 전략 교체 승인 요청
- 실전 후보 승격 승인 요청
- 자동화 확장 허용 승인 요청

### 6.2 리스크 이벤트

- 일 손실 한도 근접/초과
- 세션 누적 손실 한도 근접/초과
- MDD 초과
- 서킷 브레이커 발동
- 자산군 노출 한도 초과

### 6.3 실행 이벤트

- 주문 실패 반복
- 부분체결 반복
- 실행 직전 재검증 실패
- 세션 자동 중지
- 포지션 상태 오류

### 6.4 연구 이벤트

- 연구 작업 완료
- 상위 후보 전략 갱신
- `paper_candidate` 신규 생성
- 연구 작업 실패/재시도

### 6.5 AI 이벤트

- AI 재분석 완료
- AI 파라미터 조정 제안
- AI 전략 교체 제안
- AI 리스크 중지 권고

## 7. 기본 채널 매핑

### 7.1 Critical

- 대시보드: 필수
- 텔레그램: 필수
- 디스코드: 선택적

### 7.2 Warning

- 대시보드: 필수
- 텔레그램: 중요 항목만
- 디스코드: 요약 또는 배치 전송

### 7.3 Info

- 대시보드: 필수
- 텔레그램: 기본 비활성
- 디스코드: 요약 전송 가능

## 8. 알림 문구 원칙

### 8.1 기본 구조

- 이벤트 종류
- 대상 자산 또는 세션
- 현재 상태
- 왜 발생했는지
- 사용자가 해야 할 다음 액션

### 8.2 예시

- `승인 필요: BTC-OKX-SWAP 전략을 EMA Trend에서 Donchian Breakout으로 교체하려고 합니다. 최근 기대값 개선, MDD 안정화가 이유입니다. 승인 후 실행됩니다.`

- `위험 경고: KRW-XRP-UPBIT 페이퍼 세션이 일 손실 한도 3%에 근접했습니다. 신규 진입이 제한될 수 있습니다.`

- `연구 완료: alt_cross_sectional_momentum 전략이 paper_candidate로 승격되었습니다.`

## 9. 알림 억제와 중복 방지

### 9.1 기본 원칙

- 같은 이벤트를 짧은 시간 안에 반복 전송하지 않는다.
- 상태 변화가 없으면 중복 전송을 줄인다.

### 9.2 억제 예시

- 동일 세션 동일 경고는 일정 시간 내 1회만
- 연구 루프 진행 중간 로그는 외부 채널에 연속 전송하지 않음
- 부분체결 반복은 집계해서 전송 가능

## 10. 집계형 알림

### 10.1 일일 요약

- 오늘 손익
- 활성 세션 수
- 연구 완료 수
- 신규 후보 수
- 주요 경고 수

### 10.2 연구 요약

- 상위 후보 전략
- 승격 후보
- 실패 run 요약

### 10.3 주간 요약

- 전략별 성과
- 자산군별 성과
- 리스크 이벤트 요약
- AI 개입 이력 요약

## 11. 승인 워크플로우 연계

- 승인 요청은 단순 알림이 아니라 `행동 요청`이어야 한다.
- 승인 알림에는 반드시 아래가 포함되어야 한다.
  - 대상
  - 변경 내용
  - 이유
  - 예상 리스크
  - 승인 필요 여부

### 승인 요청 필수 채널

- 대시보드
- 텔레그램

## 12. 리스크 연계

- 리스크 이벤트는 전략 이벤트보다 우선 순위를 가진다.
- Critical 리스크 이벤트는 즉시 외부 채널로도 전달한다.
- 반복적 경고는 집계형과 즉시형을 구분한다.

## 13. 페이퍼트레이딩 연계

- 페이퍼 세션 시작/중지/실패는 알림 대상이다.
- 부분체결 반복, 슬리피지 급등, 자동 중지는 Warning 이상으로 처리한다.
- 페이퍼 통과 또는 실전 후보 승격은 Info가 아니라 승인/중요 이벤트로 취급한다.

## 14. 연구 루프 연계

- 연구 루프 진행 로그는 기본적으로 대시보드 중심
- 상위 후보 승격, 후보 탈락, 주요 실패는 외부 채널 전송 가능
- 지나치게 세밀한 중간 진행률은 외부 채널에 보내지 않는다.

## 15. AI 연계

- AI 결과는 "권고"와 "행동 요청"을 구분해서 보낸다.
- 단순 해석 결과는 Info
- 전략 교체나 세션 중지 권고는 Warning 이상

## 16. 사용자 설정 가능 항목

- 텔레그램 사용 여부
- 디스코드 사용 여부
- 채널별 이벤트 범위
- 일일 요약 수신 여부
- 연구 요약 수신 여부
- Warning/Info 필터 레벨

## 17. 저장 구조

알림 시스템은 아래를 저장해야 한다.

- notification_id
- event_type
- priority
- target_channel
- target_session_or_asset
- message_summary
- message_detail
- sent_at
- acknowledged_at
- related_approval_id

## 18. 대시보드 반영

대시보드에는 아래가 보여야 한다.

- 승인 대기 항목
- Critical 경고
- 최근 Warning
- 최근 Info 요약
- 읽지 않은 알림 수

## 19. 성공 기준

- 승인 요청과 Critical 리스크 이벤트를 놓치지 않게 전달할 수 있다.
- 대시보드와 외부 채널이 같은 이벤트 체계를 공유한다.
- 알림 과다 전송 없이 중요한 것만 우선 전달할 수 있다.
- 사용자가 다음 행동을 바로 이해할 수 있는 메시지를 제공한다.

## 20. 오픈 질문

- 텔레그램을 1차 기본 채널로 고정할지
- 디스코드는 운영 로그용으로 더 강하게 쓸지
- 승인 응답을 외부 채널에서 직접 받을지, 대시보드에서만 처리할지
