# 10 Execution Engine Spec

## 1. 문서 목적

- 전략 시그널이 실제 주문 요청, 체결 상태, 포지션 상태, 세션 상태로 어떻게 이어지는지 정의한다.
- 페이퍼트레이딩 중심으로 구현하되, 실전 확장이 가능한 동일 인터페이스 구조를 설계한다.
- 이 문서는 구현 지시용 PRD이며, 대화 과정은 남기지 않고 `합의된 정책`만 기록한다.

## 2. 한 줄 정의

- 실행 엔진은 오케스트레이터와 전략 엔진이 만든 의사결정을 받아, 사전 검증과 재검증을 거친 뒤 주문을 생성하고, 체결/부분체결/실패/중지를 일관된 상태 모델로 관리하는 계층이다.

## 3. 범위

### 포함

- 페이퍼/실전 공통 실행 인터페이스
- 전략 세션 중심 상태 관리
- 주문 생성
- 실행 직전 재검증
- 체결/부분체결/취소/재주문 처리
- 포지션 상태 동기화
- 실패/재시도 처리
- 실행 로그와 상태 이력

### 제외

- 개별 거래소 API 명세 상세
- 백테스트 엔진 내부 로직
- 리스크 한도 계산 상세
- 대시보드 UI 상세

## 4. 핵심 원칙

- 인터페이스는 `페이퍼와 실전 공통`으로 설계한다.
- 실제 구현은 `페이퍼 중심`으로 시작한다.
- 승인 후에도 `실행 직전 재검증`을 반드시 수행한다.
- 실행 엔진의 기본 관리 단위는 `전략 세션`이다.
- 주문 성향은 전략별로 다르게 적용한다.
- 부분체결은 예외가 아니라 기본적으로 발생 가능한 상태로 본다.
- 실패와 재시도는 로그와 상태 전이로 명확히 남긴다.

## 5. 실행 엔진의 역할

- 전략 또는 오케스트레이터가 만든 주문 의도를 받는다.
- 실행 가능 조건을 사전 검증한다.
- 승인 필요 시 승인 대기 상태를 만든다.
- 실행 직전에 시장 상태를 다시 점검한다.
- 주문을 발행하거나 페이퍼 주문을 시뮬레이션한다.
- 체결, 부분체결, 실패, 취소, 재주문을 상태 머신으로 관리한다.
- 최종적으로 포지션, 세션, 로그를 갱신한다.

## 6. 아키텍처 방향

### 6.1 공통 인터페이스

- `ExecutionEngine` 인터페이스는 페이퍼와 실전이 공유한다.
- 차이는 내부 adapter 구현에서만 발생한다.

### 6.2 초기 구현 방향

- V2에서는 `PaperExecutionAdapter`를 먼저 구현한다.
- 실전은 `LiveExecutionAdapter`가 붙을 수 있는 자리만 만들어둔다.

### 6.3 기대 효과

- 페이퍼에서 검증한 상태 모델과 로그 구조를 실전에도 재사용할 수 있다.
- 향후 거래소/브로커가 늘어나도 상위 로직을 유지하기 쉽다.

## 7. 관리 단위

### 7.1 전략 세션 중심

실행 엔진의 기본 관리 단위는 `전략 세션`이다.

하나의 세션은 아래를 묶는다.

- 전략
- 자산 또는 자산 바스켓
- 운용 모드
- 리스크 정책
- 현재 포지션
- 주문 이력
- 승인 상태

### 7.2 이유

- 단일 주문 중심으로 보면 전략 단위 문맥이 끊긴다.
- 포지션 중심으로만 보면 승인, 교체, 전략 변경 이력이 약해진다.
- 전략 세션 중심 구조가 오케스트레이터, 페이퍼, 리스크, 대시보드와 가장 잘 연결된다.

## 8. 상태 모델

### 8.1 세션 상태

- `draft`
- `approval_pending`
- `ready`
- `running`
- `paused`
- `stop_requested`
- `stopped`
- `failed`
- `completed`

### 8.2 주문 상태

- `pending_validation`
- `pending_approval`
- `approved`
- `rejected`
- `queued`
- `submitted`
- `partially_filled`
- `filled`
- `cancel_requested`
- `cancelled`
- `replaced`
- `failed`

### 8.3 포지션 상태

- `flat`
- `opening`
- `open`
- `reducing`
- `closing`
- `closed`
- `error`

## 9. 실행 흐름

### 9.1 기본 흐름

1. 전략이 주문 의도를 생성한다.
2. 실행 엔진이 사전 검증을 수행한다.
3. 승인 필요 시 `approval_pending`으로 전환한다.
4. 승인 완료 후 실행 직전 재검증을 수행한다.
5. 주문을 발행하거나 페이퍼 체결 시뮬레이션을 수행한다.
6. 체결/부분체결/실패에 따라 포지션과 세션 상태를 갱신한다.
7. 로그와 이벤트를 저장한다.

### 9.2 실행 직전 재검증

승인 직후라도 아래 조건을 다시 확인한다.

- 가격 급변 여부
- 유동성 급감 여부
- warning/caution 상태
- 서킷 브레이커 상태
- 슬리피지 급등 상태
- 장외/휴장/거래불가 상태

### 9.3 재검증 실패 시 정책

- 주문 실행을 취소한다.
- 필요 시 `재승인 필요` 상태로 되돌린다.
- 사용자에게 이유를 알린다.

## 10. 주문 성향 정책

### 10.1 기본 원칙

- 주문 성향은 `전략별`로 다르게 가져간다.
- 모든 전략에 동일한 체결 정책을 쓰지 않는다.

### 10.2 기본값

#### 추세추종 / 브레이크아웃

- 기본 성향: `체결 우선`
- 이유:
  - 진입 타이밍을 놓치면 전략 논리가 무너질 수 있다.

#### 평균회귀

- 기본 성향: `가격 우선`
- 이유:
  - 불리한 가격 추격은 기대값을 빠르게 훼손할 수 있다.

#### 실행 정책 메타데이터

전략은 아래와 같은 실행 성향 메타데이터를 가져야 한다.

- `fill_priority`
  - `execution_first`
  - `price_first`
- `preferred_order_type`
  - `market`
  - `limit`
  - `best`
- `max_requote_count`
- `max_wait_ms`

## 11. 부분체결 정책

### 11.1 기본 정책

- 부분체결 발생 시 기본은 `일정 시간 대기 후 재주문`
- 고위험 이벤트 구간은 `잔량 취소`

### 11.2 일반 구간 처리

- 일부 체결 후 미체결 잔량은 대기 상태로 둔다.
- 설정된 시간 또는 틱 수 경과 후 다시 주문 여부를 판단한다.
- 재주문 시 가격/수량/실행 성향을 다시 계산할 수 있다.

### 11.3 고위험 이벤트 구간 처리

대상 예시:

- 9시 이벤트 구간
- warning/caution
- 청산 급증 구간
- 장 시작/장 마감 급변 구간
- 유동성 급감

정책:

- 미체결 잔량은 기본적으로 취소
- 무리한 시장가 전환은 피한다.

### 11.4 왜 이렇게 하나

- 무조건 시장가 전환하면 고위험 구간에서 슬리피지가 급격히 커질 수 있다.
- 무조건 대기하면 추세 전략의 타이밍이 무너질 수 있다.
- 따라서 기본은 `재주문`, 고위험 시 `취소`가 V2에 가장 균형적이다.

## 12. 재주문 정책

### 12.1 재주문 조건

- 부분체결 잔량 존재
- 세션이 계속 유효
- 리스크 한도 미초과
- 재검증 통과

### 12.2 재주문 시 조정 가능 항목

- 주문 가격
- 주문 수량
- 주문 타입
- 타임인포스

### 12.3 재주문 제한

- 전략 메타데이터의 `max_requote_count`를 넘기지 않는다.
- 제한 초과 시 잔량 취소 또는 세션 경고 처리

## 13. 실패 처리

### 13.1 실패 유형

- validation failure
- approval timeout
- revalidation failure
- submission failure
- cancel failure
- state sync failure
- repeated partial fill failure

### 13.2 기본 정책

- 실패는 반드시 상태와 로그에 남긴다.
- 반복 실패 시 세션을 `paused`로 전환할 수 있다.
- 동일 원인 실패가 누적되면 오케스트레이터와 리스크 계층에 전달한다.

## 14. 포지션 동기화

### 14.1 포지션은 주문 결과의 부산물이 아니라 핵심 상태다.

- 포지션 상태는 주문 상태와 별도로 관리한다.
- 부분체결 상태에서도 포지션은 `opening`, `reducing`, `closing` 등을 가질 수 있다.

### 14.2 동기화 원칙

- 주문 이벤트 발생 시 포지션 상태를 즉시 갱신한다.
- 세션 기준으로 최종 포지션이 무엇인지 항상 복원 가능해야 한다.
- 포지션 상태와 주문 로그가 충돌하면 오류로 기록한다.

## 15. 승인 흐름

### 15.1 승인 필요 상황

- 신규 세션 시작
- 전략 교체
- 고위험 전략 배치
- 실전 후보 승격

### 15.2 승인 이후

- 승인 즉시 주문하지 않는다.
- 실행 직전 재검증을 수행한다.

### 15.3 승인 무효화 조건

- 가격 급변
- 리스크 상태 악화
- 시장 이벤트 발생
- 데이터 이상

## 16. 페이퍼 / 실전 공통 인터페이스

실행 엔진은 아래 인터페이스를 기준으로 통일한다.

### 16.1 필수 공통 메서드 예시

- `validateOrderIntent`
- `requestApproval`
- `revalidateBeforeExecution`
- `submitOrder`
- `cancelOrder`
- `replaceOrder`
- `syncPositionState`
- `pauseSession`
- `stopSession`

### 16.2 Paper Adapter

- 실제 주문 대신 시뮬레이션 체결
- 슬리피지/부분체결/실패 모델 적용

### 16.3 Live Adapter

- 실거래 API 호출
- 실체결/실포지션 동기화
- V2에서는 인터페이스만 설계

## 17. 로그 및 저장 구조

실행 엔진은 아래를 저장해야 한다.

- session log
- order intent
- validation result
- approval event
- revalidation result
- order event
- fill event
- cancel/replace event
- position state history
- execution error history

## 18. 오케스트레이터 연계

- 실행 엔진은 오케스트레이터가 내린 배치 결정을 수행 계층으로 전달받는다.
- 실행 중 실패, 반복 부분체결, 재검증 실패는 다시 오케스트레이터에 피드백한다.
- 오케스트레이터는 이 실행 품질 정보를 전략 점수와 유지/교체 판단에 반영한다.

## 19. 리스크 관리 연계

- 리스크 계층은 실행 엔진보다 상위 우선순위를 가진다.
- 리스크 한도 초과 시 주문 실행을 차단할 수 있다.
- 실행 직전 재검증에도 리스크 상태가 반드시 포함되어야 한다.

## 20. 대시보드 노출 항목

- 세션 상태
- 승인 대기 상태
- 주문 진행 상태
- 부분체결 상태
- 재주문 횟수
- 최근 실패 사유
- 재검증 실패 사유
- 포지션 상태

## 21. 성공 기준

- 동일 인터페이스로 페이퍼/실전 확장 구조를 유지할 수 있다.
- 실행 엔진의 기본 관리 단위가 전략 세션으로 작동한다.
- 승인 후 재검증이 반드시 수행된다.
- 부분체결 시 기본 정책이 일관되게 적용된다.
- 전략별 주문 성향을 메타데이터로 제어할 수 있다.
- 실패와 상태 전이가 모두 복원 가능하게 저장된다.

## 22. 오픈 질문

- 재검증 유효 시간 창을 몇 초/몇 분으로 둘지
- 전략별 기본 `max_requote_count`를 얼마로 둘지
- 승인 만료 시간을 얼마로 둘지
- 장 시작/9시 이벤트 같은 고위험 구간 정의를 어느 문서에 최종 고정할지
