# 02 Orchestrator Spec

## 1. 문서 목적

- 오케스트레이터가 무엇을 판단하고 무엇을 하지 않는지 정의한다.
- 자산별 전략 배치, 교체, 정지, 승격의 기준을 정리한다.
- AI 개입 조건, 승인 워크플로우, 자동/수동 운용 모드를 명확히 나눈다.
- 이후 데이터 아키텍처, 연구 루프, 페이퍼트레이딩, 대시보드 명세의 기준점으로 사용한다.

## 2. 한 줄 정의

- 오케스트레이터는 `각 자산 또는 자산 바스켓에 어떤 전략을 어떤 모드로 배치할지`를 결정하고, 시장 변화와 성과 변화를 감시하며, 필요 시 유지/교체/정지/재분석을 수행하는 상위 제어 계층이다.

## 3. 범위

### 포함

- 자산군별 전략 배치
- 전략 후보 점수화
- 전략 유지/교체/정지 판단
- 연구 루프 결과의 승격 판단
- 페이퍼트레이딩 배포 판단
- AI 호출 이벤트 정의
- 승인 필요 상황 정의
- 수동/반자동/자동 운용 모드 정의
- 자산별 전략 배치 상태 저장
- 자산/전략/상태를 대시보드에 노출하기 위한 상태 모델 정의

### 제외

- 백테스트 내부 계산 로직 상세
- 주문 체결 엔진 구현 상세
- 거래소 API 호출 상세
- 대시보드 UI 레이아웃 상세
- 사용자 인증/권한 모델 상세

## 4. 적용 범위

### 지원 자산군

- 암호화폐
- 한국주식

### 우선 채널

- Upbit 현물
- OKX 선물
- 국내 증권 API 기반 한국주식 데이터

### 운용 범위

- 암호화폐: 백테스트 + 페이퍼트레이딩 우선
- 한국주식: 백테스트 + 페이퍼트레이딩 우선
- 실전 자동매매: 구조는 고려하되 핵심 범위는 아님

## 5. 오케스트레이터의 역할

오케스트레이터는 단순히 "좋아 보이는 전략 하나"를 고르는 시스템이 아니다.

- 자산별 시장 상태를 해석한다.
- 해당 자산에 적용 가능한 전략 후보군을 불러온다.
- 전략별 적합도와 최근 검증 성과를 점수화한다.
- 현재 전략을 유지할지, 다른 전략으로 교체할지, 잠시 중지할지 판단한다.
- 성과 악화, 시장 구조 변화, 리스크 이벤트가 생기면 재평가를 실행한다.
- 페이퍼트레이딩에서는 자동 조정할 수 있다.
- 실전에서는 승인 요청을 생성하고 자동 집행은 하지 않는다.

### 5.1 연구 오케스트레이션

연구 오케스트레이션은 `무엇이 좋은 전략인지 검증 데이터를 만드는 계층`이다.

- 전략 후보 생성
- 지표/파라미터 자동 탐색
- 백테스트 실행
- 워크포워드 검증
- 결과 저장 및 랭킹
- 페이퍼 후보 승격 대상 생성

연구 오케스트레이션의 산출물은 아래와 같다.

- 전략별 백테스트 결과
- 비용 반영 후 기대값
- 총수익, MDD, 샤프/고급 통계
- 파라미터 조합별 비교 결과
- 과최적화 방지용 검증 결과

### 5.2 운용 오케스트레이션

운용 오케스트레이션은 `이미 검증된 데이터와 현재 시장 상태를 바탕으로 배치 결정을 내리는 계층`이다.

- 자산별 후보 전략 선택
- 전략 유지/교체/중지 판단
- 페이퍼 배포 판단
- 승인 요청 생성
- 리스크 이벤트 대응
- 대시보드용 상태 제공

### 5.3 핵심 전제

- 운용 오케스트레이션은 연구 오케스트레이션이 만든 검증 데이터 위에서만 동작한다.
- 백테스트 검증 데이터가 없는 전략은 페이퍼트레이딩으로 승격하지 않는다.
- 페이퍼트레이딩 데이터가 충분히 쌓이지 않은 전략은 실전 후보로 승격하지 않는다.
- 즉, 오케스트레이터는 `근거 없는 전략 선택기`가 아니라 `검증 데이터 기반 전략 배치기`여야 한다.

## 6. 핵심 개념

### 6.1 배치 단위

- `asset_slot`
  - 단일 자산 단위
  - 예: `BTC-OKX-SWAP`, `KRW-XRP-UPBIT`, `KRX-005930`
- `basket_slot`
  - 여러 자산을 묶어 하나의 전략이 적용되는 단위
  - 예: `UPBIT_ALT_TOP20`, `KOREAN_STOCK_MOMENTUM_POOL`
- `global_filter_slot`
  - 시장 전체 또는 자산군 전체에 영향을 주는 필터
  - 예: `crypto_risk_guard`, `kimchi_premium_filter`

### 6.2 전략 역할 유형

- `primary_strategy`
  - 실제 진입/청산 의사결정을 만드는 핵심 전략
- `filter_strategy`
  - 전략 활성/비활성 여부를 결정하는 필터
- `risk_guard`
  - 손실 제한, 거래 중지, 비중 축소 같은 안전장치
- `execution_policy`
  - TWAP/VWAP, 지정가/시장가 정책 등 체결 방식
- `ranking_signal`
  - 직접 매매는 하지 않지만 오케스트레이터의 점수 계산에 쓰이는 신호

### 6.3 배치 상태

- `research_only`
- `paper_candidate`
- `paper_running`
- `paper_paused`
- `live_candidate`
- `approval_pending`
- `live_running`
- `retired`

### 6.4 오케스트레이션 사이클

- 일정 주기 또는 이벤트 발생 시 수행되는 재평가 루프
- 예:
  - 1시간마다 자산별 전략 적합도 재평가
  - 새 백테스트 결과가 저장되면 후보 재정렬
  - warning/caution 발생 시 즉시 재평가

## 7. 운용 모드

### 7.1 수동 모드

- 사용자가 전략을 직접 선택한다.
- 오케스트레이터는 추천과 감시만 수행한다.
- 자동 교체는 하지 않는다.
- 단, 리스크 이벤트는 강하게 경고한다.

### 7.2 반자동 모드

- 오케스트레이터가 전략 변경 또는 승격을 제안한다.
- 사용자가 승인하면 반영된다.
- 초기 기본 모드는 이 방식을 따른다.
- 장기적으로 실전 운용의 기본 모드도 이쪽이 된다.

### 7.3 자동 모드

- 오케스트레이터가 전략 배치와 교체를 자동 수행한다.
- 초기에는 기본값으로 두지 않는다.
- 충분한 검증 데이터와 운영 안정성이 쌓인 뒤 `페이퍼트레이딩`에서 우선 허용한다.
- 실전에서는 직접 자동 집행이 아니라 승인 요청 생성까지로 제한한다.

## 8. 자산별 배치 원칙

### 8.1 기본 원칙

- 하나의 `asset_slot`에는 기본적으로 `주 전략 1개`만 활성화한다.
- 여기에 `필터 0개 이상`, `리스크 가드 0개 이상`을 덧붙인다.
- `basket_strategy`는 별도 슬롯으로 관리한다.
- 동일 자산에 상충되는 두 개의 `primary_strategy`를 동시에 활성화하지 않는다.

### 8.2 자산군별 예시

- BTC 선물
  - `primary_strategy`: 추세추종 또는 평균회귀 중 1개
  - `filter_strategy`: 시장 구조 필터, 펀딩/변동성 필터
  - `risk_guard`: 레버리지 제한, 서킷 브레이커
- 업비트 알트
  - `primary_strategy`: 유동성 필터 추세, 눌림매수, 평균회귀 등 1개
  - `filter_strategy`: warning/caution, 김프 필터, 거래대금 조건
  - `risk_guard`: 상장폐지/유의 이벤트 대응
- 한국주식
  - 초기에는 페이퍼 전용
  - 전략군과 필터를 분리하되, 실전 승인/집행은 후순위

## 9. 전략 등록 계약

오케스트레이터에서 전략을 자유롭게 추가하려면, 전략은 공통 메타데이터를 반드시 가져야 한다.

### 9.1 필수 메타데이터

| 필드 | 설명 |
|---|---|
| `strategy_id` | 전략 고유 식별자 |
| `name` | 표시 이름 |
| `asset_class` | `crypto_futures`, `crypto_spot`, `korean_stock` 등 |
| `deployment_type` | `single_asset`, `basket`, `filter`, `risk_guard`, `execution_only` |
| `direction` | `long_only`, `long_short` |
| `supported_markets` | Upbit, OKX, KRX 등 |
| `required_data` | OHLCV, orderbook, funding, OI, warning flag 등 |
| `timeframe` | 1m, 15m, 1h, 4h, 1d 등 |
| `risk_profile` | `low`, `medium`, `high` |
| `cost_sensitivity` | `low`, `medium`, `high` |
| `compatible_modes` | `manual`, `semi_auto`, `auto_paper`, `auto_live_candidate` |
| `minimum_history_requirement` | 필요한 최소 검증 기간 |
| `minimum_trade_requirement` | 최소 거래 수 |
| `supports_backtest` | 백테스트 가능 여부 |
| `supports_paper` | 페이퍼 가능 여부 |
| `supports_live_candidate` | 실전 후보 승격 가능 여부 |

### 9.2 선택 메타데이터

- 추천 시장 구조
- 금지 시장 구조
- 권장 자산군
- 상위 필터 의존성
- 설명용 텍스트
- 사용자 난이도

### 9.3 등록 거절 조건

- 필수 메타데이터가 없음
- 필요한 데이터 소스를 현재 시스템이 지원하지 않음
- 백테스트 또는 페이퍼 검증 경로가 없음
- 리스크 프로필이 정의되지 않음

### 10.1 공통 입력

- 가격 추세

- 전략별 최근 손익비

## 11. 자산 유니버스 선정 규칙

### 11.1 암호화폐 선물

- 우선 유니버스:
  - BTC
  - ETH
- 후속 확장 시:
  - 유동성, 수수료, 슬리피지 모델이 확보된 자산만 추가

### 11.2 업비트 알트 현물

- 기본 유니버스:
  - Upbit 거래 가능 종목 중 정책상 제외되지 않은 종목
- 필수 필터:
  - 거래대금 최소 조건
  - 스프레드 상한
  - warning/caution 상태 제외 또는 제한
  - 데이터 결측 심한 종목 제외

### 11.3 한국주식

- V2에서는 종목 풀을 좁게 시작
- 예:
  - 대형주 중심
  - 거래대금 충분한 종목 중심
- 저유동성 종목은 후순위

## 12. 전략 후보 생성 방식

오케스트레이터는 모든 전략을 모든 자산에 무차별 적용하지 않는다.

### 12.1 전략 후보군 생성 순서

1. 자산군별 호환 전략만 불러온다.
2. 필수 데이터가 있는 전략만 남긴다.
3. 현재 운용 모드에서 허용된 전략만 남긴다.
4. 현재 시장 구조와 충돌하는 전략을 제외한다.
5. 최근 검증 성과가 최소 기준 미달인 전략을 제외한다.

### 12.2 예시

- BTC 선물 + 강한 추세 + 높은 변동성
  - 추세 전략 우선
  - 평균회귀 전략 점수 하향
- 업비트 알트 + warning 발생
  - 공격적 돌파 전략 제외
  - 리스크 필터 우선
- 한국주식 + 데이터 부족
  - 전략 실행 불가
  - 관찰 상태 유지

## 13. 전략 점수 산정 규칙

점수는 단순 수익률 한 가지로 정하지 않는다.

### 13.0 평가 우선순위

V2에서 오케스트레이터가 전략을 평가할 때의 우선순위는 아래와 같다.

1. 비용 반영 후 기대값이 양수인가
2. 총수익이 의미 있게 남는가
3. MDD가 감당 가능한 수준인가
4. 다른 구간에서도 성과가 무너지지 않는가
5. 승률은 보조 지표로 참고하는가

승률은 높아도 큰 손실 한 번으로 기대값이 무너질 수 있으므로, 단독 기준으로 사용하지 않는다.

### 13.1 점수 구성

| 항목 | 설명 | 기본 비중 |
|---|---|---|
| `market_fit_score` | 현재 시장 구조와 전략의 적합성 | 20 |
| `validation_score` | 최근 백테스트/워크포워드/페이퍼 검증 품질 | 20 |
| `performance_score` | 최근 기대값, 총수익, 손익비 중심 성과 | 25 |
| `risk_score` | MDD, 꼬리위험, 리스크 안정성 | 20 |
| `execution_score` | 체결 가능성, 비용 민감도, 데이터 충족도 | 10 |
| `policy_score` | 사용자 선호, 허용 리스크, 모드 적합성 | 5 |

총합은 `0-100` 기준으로 정규화한다.

### 13.2 감점 요인

- 데이터 결측
- warning/caution 발생
- 최근 페이퍼 성과 악화
- API 지연/실행 불안정
- 과최적화 의심
- 거래 수 부족

### 13.3 점수 해석

- `80 이상`
  - 강한 후보
  - 자동 페이퍼 배치 또는 유지 가능
- `65-79`
  - 유효 후보
  - 관찰 또는 반자동 추천 대상
- `50-64`
  - 보류
  - 추가 검증 필요
- `50 미만`
  - 배치 금지 또는 일시 중지

## 14. 하드 게이트 규칙

점수 계산 전에 아래 조건을 먼저 통과해야 한다.

- 필요한 데이터 소스가 정상 동작해야 한다.
- 최소 검증 기간이 확보되어야 한다.
- 최소 거래 수가 충족되어야 한다.
- 정책상 금지된 자산/전략 조합이 아니어야 한다.
- warning/caution 등 치명적 리스크 상태가 아니어야 한다.
- 리스크 한도 초과 전략은 자동 배치 대상에서 제외한다.

하드 게이트를 통과하지 못하면 점수와 관계없이 `inactive` 처리한다.

### 14.1 페이퍼 후보 진입 게이트

아래는 `paper_candidate`로 올리기 위한 기본 확정값이다.

- 백테스트가 완료되어 있어야 한다.
- 백테스트는 `in-sample`만이 아니라 `out-of-sample` 또는 워크포워드 검증을 포함해야 한다.
- 백테스트에는 수수료, 슬리피지, 펀딩비 같은 현실 비용이 반영되어야 한다.
- 비용 반영 후 기대값이 `0보다 커야` 한다.
- 비용 반영 후 총수익이 `0보다 커야` 한다.
- MDD가 전략 슬롯 허용 한도 이하이어야 한다.
- 최소 표본을 충족해야 한다.

#### 최소 표본 기본값

- 저빈도 전략:
  - 최소 `180일` 백테스트
  - 최소 `30회` 청산 완료 거래
- 중빈도 전략:
  - 최소 `90일` 백테스트
  - 최소 `50회` 청산 완료 거래
- 고빈도/초단타 전략:
  - 최소 `30일` 고해상도 백테스트
  - 최소 `200회` 청산 완료 거래

#### 고급 검증 권장 항목

- PBO 또는 유사 과최적화 검증
- DSR 또는 PSR 같은 다중 테스트 보정 지표

초기에는 위 고급 검증을 `권장`으로 두고, 시스템이 안정화되면 `필수 게이트`로 승격한다.

### 14.2 페이퍼 통과 게이트

아래는 `paper_running`에서 `live_candidate` 또는 상위 승인 단계로 올리기 위한 기본 확정값이다.

- 최소 `30일` 이상 페이퍼트레이딩을 수행해야 한다.
- 그리고 최소 `50회` 이상 청산 완료 거래를 만들어야 한다.
- 위 두 조건은 `둘 중 하나`가 아니라 `둘 다` 충족하는 것을 기본으로 한다.
- 페이퍼 기간 동안 비용 반영 후 기대값이 양수여야 한다.
- 페이퍼 기간 동안 총손익이 양수여야 한다.
- 페이퍼 MDD가 전략별 허용 범위를 넘지 않아야 한다.
- 백테스트 대비 성과가 구조적으로 붕괴하지 않아야 한다.
- 주문 실패, 데이터 결측, 비정상 체결 같은 운영 오류가 반복되지 않아야 한다.

#### 백테스트 대비 열화 허용 기본값

- 페이퍼 기대값이 백테스트 기대값의 `50% 미만`으로 떨어지면 경고
- 페이퍼 기대값이 `0 이하`면 통과 불가
- 실제 슬리피지가 모델 가정보다 지속적으로 큰 경우 재검증

#### 전략별 기본 MDD 가이드

- 저위험 전략: `10% 이하`
- 중위험 전략: `15% 이하`
- 고위험 전략: `20% 이하`

위 수치는 기본 확정값이며, 사용자 리스크 성향과 실제 운영 데이터에 맞춰 추후 조정 가능하다.

## 15. 주요 의사결정 액션

오케스트레이터는 재평가 후 아래 액션 중 하나를 선택한다.

- `retire_strategy`
  - 더 이상 사용하지 않음
- `request_ai_review`

- `request_manual_approval`
  - 사용자 승인 요청 생성

## 17. AI 호출 이벤트

AI는 상시 실행하지 않고, 특정 이벤트가 생겼을 때만 호출한다.

### 17.1 호출 조건

- 성과 악화 원인 분석 필요

### 17.2 AI 입력

- 자산 정보

- 시장 구조 신호
- 운영 상태 정보

### 17.3 AI 출력

- 유지 / 교체 / 중지 추천
- 추천 이유 요약
- 조정이 필요한 파라미터 후보
- 추가 검증이 필요한 항목
- 승인 필요 여부

### 17.4 AI 권한

- 페이퍼트레이딩: 자동 조정 가능

## 19. 전략 승격 단계

V2에서는 전략이 아래 단계를 거쳐 이동한다.

1. `research_only`
   - 아이디어 또는 리서치 단계
2. `backtest_ready`
   - 구조상 백테스트 가능
3. `validated_candidate`
   - 기본 검증 기준 통과
4. `paper_candidate`
   - 페이퍼 배치 후보
5. `paper_running`
   - 실제 페이퍼트레이딩 중
6. `paper_verified`
   - 일정 기간 성과와 안정성 검증 완료
7. `live_candidate`
   - 실전 후보
8. `approval_pending`
   - 사용자 승인 대기
9. `live_running`
   - 실전 운용 중
10. `retired`
   - 폐기

### 19.1 승격 금지 원칙

- 백테스트 검증 데이터가 없는 전략은 `paper_candidate`로 이동할 수 없다.
- 페이퍼 검증이 부족한 전략은 `live_candidate`로 이동할 수 없다.
- 높은 총수익이 나와도 MDD와 운영 실패가 기준을 넘으면 승격하지 않는다.
- 승률만 높고 기대값이 음수인 전략은 승격하지 않는다.

## 22. 성공 기준

- 자산별로 전략을 따로 배치할 수 있다.
- 동일 시스템 안에서 수동/반자동/자동 모드를 구분할 수 있다.
- 백테스트/페이퍼 결과를 기반으로 전략 후보를 정렬할 수 있다.
- 성과 악화와 시장 구조 변화에 따라 전략 교체를 판단할 수 있다.
- 페이퍼에서는 자동 조정, 실전에서는 승인 요청 생성이 가능하다.
- 전략 추가 시 공통 메타데이터만 맞추면 후보군에 편입할 수 있다.
